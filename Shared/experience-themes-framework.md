## 一、框架总览

### 1.1 为什么需要体验主题框架

传统的功能模块分析（圣经模块、计划模块、小组模块、祷告模块、资源模块）关注的是"我们建了什么"。但用户并不按模块思考——他们带着意图来，期待顺畅地完成任务。当意图跨越多个模块时，功能视角容易产生盲区：每个模块看起来都"做了该做的事"，但用户的端到端旅程却是断裂的。

体验主题框架从用户意图出发，追踪"用户来干什么、体验如何、哪里卡住了"。它是功能模块分析的互补视角，不是替代。

### 1.2 产品竞争定位：属灵空间而非信息工具

在AI时代，通用AI工具（ChatGPT、Gemini等）已经能完成许多传统圣经软件的功能：经文检索、背景解释、主题汇总、甚至生成祷告文。但圣经App提供的不仅是信息服务，更是**属灵空间**：

- **仪式感**：一个专属的、安静的信仰操练环境，不同于通用聊天窗口
- **关系容器**：小组代祷、共读计划、感想分享——这些需要持久的社群关系，不是一次性的AI对话
- **结构化信仰根基**：注释体系、原文工具、交叉引用——专业领域知识图谱，AI的准确性和权威性尚不够可靠
- **属灵习惯养成**：读经计划、祷告提醒、连续打卡——需要时间维度的持续性，不是单次查询
- **心理安全**：在这个空间里写下的祷告和感想是私密的、安全的，不是被训练AI的语料

因此，产品竞争的不是信息效率，而是**信仰体验的深度和持续性**。体验主题框架正是围绕这个定位展开的。

### ET-01 注册/登录/游客

**核心问题**：如何以最低摩擦建立用户身份，同时为后续个人化体验打好基础？

#### 用户场景

| 场景 | 用户画像 | 触发情境 | 目标 |
|------|---------|---------|------|
| S1-1 | 教会弟兄姐妹收到同工推荐 | 应用商店下载后首次打开 | 快速开始使用，不想在注册上花太多时间 |
| S1-2 | 小组成员收到组长的邀请链接 | 点击微信分享链接 | 通过链接直接加入小组，希望过程顺畅 |
| S1-3 | 好奇的慕道友自行搜索下载 | 应用商店搜索"圣经" | 先看看有什么，不想注册就被要求"交出"个人信息 |
| S1-4 | 老用户换新手机 | 新设备上重新下载App | 恢复之前所有数据（计划进度、笔记、祷告记录） |
| S1-5 | 用户在多台设备间切换 | 手机看完想在平板继续 | 数据实时同步，体验无缝衔接 |

#### 当前产品能力

> ⚠️ 以下为基于现有文档的推测，尚未通过截图验证，需要实际确认

- 支持的登录方式：待确认（推测包含手机号、微信登录）
- 游客模式：待确认（推测圣经阅读基础功能可免登录使用）
- 多设备同步：待确认
- 账号恢复机制：待确认
- 邀请链接处理：`design-principles-navigation.md § 深链接` 有定义，但实际体验待验证

#### 体验断点

| 编号 | 断点描述 | 严重程度 | 说明 |
|------|---------|---------|------|
| ET01-G1 | 游客可体验范围不明确 | ⚠️ 待确认 | 不确定哪些功能需要登录、哪些不需要；如果登录门槛出现得太早，会在用户理解产品价值之前就流失 |
| ET01-G2 | 邀请链接→注册→加入小组的链路可能断裂 | ⚠️ 待确认 | 从微信链接到App内加入小组，涉及跨应用跳转，在Android/iOS上行为不同，中间有多个流失节点 |
| ET01-G3 | 换设备后的数据迁移体验 | ⚠️ 待确认 | 用户的读经进度、笔记、祷告记录、小组关系是否完整恢复 |
| ET01-G4 | 注册方式对华人用户的适配性 | ⚠️ 待确认 | 海外华人可能无法使用国内手机号注册；国内用户可能依赖微信登录；Apple登录在iOS上是否支持 |
| ET01-G5 | 登录状态管理 | ⚠️ 待确认 | 长时间不使用后的session过期处理；重新登录后是否恢复到上次位置 |

#### 竞品参考

| 竞品 | 做法 | 值得借鉴的点 |
|------|------|------------|
| **YouVersion** | 支持游客模式浏览圣经，在首次想保存高亮/书签时提示注册；支持Google、Facebook、Apple登录 | 延迟注册门槛到"用户产生保存意图"的时刻，转化率更高 |
| **Logos Bible** | 强制注册才能使用（因为涉及付费内容授权），但提供免费内容吸引注册 | 如果产品有付费内容，注册门槛的位置需要和商业模式匹配 |
| **微信读书** | 微信一键登录（利用超级App生态），登录后立即可用 | 华人用户对微信登录的接受度非常高，几乎是零摩擦 |

#### AI时代的挑战与机会

- **挑战**：通用AI工具无需注册就能回答圣经问题，用户可能质疑"为什么要在这里注册"
- **机会**：正因为这是一个需要积累数据的属灵空间（祷告记录、读经笔记、小组关系），注册是创造长期价值的前提。关键是让用户在注册前就体验到"这个空间值得我留下来"
- **策略**：游客模式应该足够好用，让用户体验到核心价值（至少能流畅阅读圣经），然后在用户想"保存"或"分享"的自然时刻引导注册

#### 关联模块

| 模块 | 关联方式 |
|------|---------|
| 所有模块 | 注册/登录是所有个人化功能的前置依赖 |
| 小组模块 | 邀请链接→注册→加入小组的完整链路 |
| 圣经模块 | 游客模式下的核心可用体验 |

#### 待澄清问题

| 编号 | 问题 | 优先级 | 需要谁来回答 |
|------|------|--------|------------|
| Q01-1 | 当前支持哪些登录方式？各方式的用户占比如何？ | P0 | 后端/数据团队 |
| Q01-2 | 游客模式下，用户可以使用哪些功能？限制策略是什么？ | P0 | 产品经理 |
| Q01-3 | 是否有注册到首次使用的漏斗数据？在哪个步骤流失最严重？ | P0 | 数据团队 |
| Q01-4 | 邀请链接在iOS和Android上的行为是否一致？ | P1 | 客户端开发 |
| Q01-5 | 是否支持账号删除（GDPR/隐私法规要求）？ | P1 | 后端/法务 |
| Q01-6 | 海外华人用户的登录方式偏好是什么？ | P2 | 用户研究 |

### ET-02 首次安装Onboarding

**核心问题**：如何让第一次使用App的用户在最短时间内理解产品价值，并找到自己的起点？

#### 用户场景

| 场景 | 用户画像 | 触发情境 | 目标 |
|------|---------|---------|------|
| S2-1 | 带着明确任务来的用户 | 组长说"下载这个App，我们一起读经" | 快速完成安装→找到小组→开始读经，不需要"教程" |
| S2-2 | 被推荐但没有明确任务 | 朋友说"这个App不错，你试试" | 了解这个App能做什么，找到和自己信仰阶段匹配的功能 |
| S2-3 | 自主搜索下载的慕道友 | 对基督教感兴趣，想读圣经 | 从零开始，需要引导"如何开始读圣经" |
| S2-4 | 从竞品切换过来的用户 | 之前用YouVersion，想试试中文体验更好的App | 了解差异化功能，快速迁移使用习惯 |

#### 当前产品能力

> ⚠️ 基于 `group-product-vision.md` 中"新手引导"作为P0待开发项的信息，推测当前产品缺乏系统性Onboarding

- 首次打开的引导流程：待确认（推测可能只有简单的欢迎页或直接进入主界面）
- 基于用户角色的差异化引导：未见相关设计
- 新手任务/快速体验路径：未见相关设计

#### 体验断点

| 编号 | 断点描述 | 严重程度 | 说明 |
|------|---------|---------|------|
| ET02-G1 | 首次打开后缺乏价值传达 | P0 | 用户可能直接面对功能列表，不知道"这个App和其他圣经App有什么不同" |
| ET02-G2 | 没有基于用户意图的分流引导 | P1 | 想读经的用户、想加入小组的用户、想祷告的用户，首次打开后看到的是同一个界面 |
| ET02-G3 | 缺少"快速体验价值"的路径 | P1 | 没有"一分钟内完成第一件有意义的事"的设计，用户可能在探索中失去耐心 |
| ET02-G4 | 从邀请链接进入的用户没有特殊引导 | P0 | 通过小组邀请链接来的用户应该直接看到小组上下文，而不是通用的产品介绍 |
| ET02-G5 | 慕道友缺乏适当的起点建议 | P2 | 信仰新人面对"创世纪到启示录"可能不知从何开始，需要推荐入门读经计划 |

#### 竞品参考

| 竞品 | 做法 | 值得借鉴的点 |
|------|------|------------|
| **YouVersion** | 3步Onboarding：选择语言→选择圣经译本→推荐"每日经文"，30秒可完成；基于邀请来源差异化引导 | 极简的初始设置，快速让用户看到内容 |
| **Headspace**（冥想App） | 首次使用引导用户完成一次3分钟冥想体验，用实际体验而非文字说明传达价值 | 让用户"做一次"比"看介绍"有效得多——可以借鉴让用户首次完成一段简短的灵修或祷告 |
| **Duolingo** | 首次使用不需要注册就能完成一节课，体验到学习的成就感后再引导注册 | 先体验后注册的策略——与ET-01的游客模式配合 |

#### AI时代的挑战与机会

- **挑战**：用户可能期待AI式的个性化推荐（"你是什么信仰阶段？我帮你定制计划"），传统的静态Onboarding显得过时
- **机会**：可以在Onboarding中引入轻量级的个性化匹配——问2-3个问题（信仰阶段、想做什么、是否有小组），然后推荐匹配的起点。这比通用AI更贴心，因为是在产品语境内的结构化引导
- **差异化方向**：AI能回答问题，但不能帮你"开始一段持续的读经习惯"——Onboarding的核心价值是帮用户建立第一个"每日回来的理由"

#### 关联模块

| 模块 | 关联方式 |
|------|---------|
| 圣经模块 | 推荐第一个读经起点（如译本选择、入门计划） |
| 计划模块 | 推荐第一个读经计划 |
| 小组模块 | 邀请链接用户的直达引导（参见 `group-experience-gaps.md § 2.6, 2.7`） |
| 祷告模块 | 可选的"首次祷告体验" |

#### 待澄清问题

| 编号 | 问题 | 优先级 | 需要谁来回答 |
|------|------|--------|------------|
| Q02-1 | 当前用户首次安装到首次"有意义操作"的平均时间是多少？ | P0 | 数据团队 |
| Q02-2 | 7日留存率是多少？主要在哪一天流失？ | P0 | 数据团队 |
| Q02-3 | 通过邀请链接安装的用户占比多少？和自然安装用户的留存差异？ | P1 | 数据团队 |
| Q02-4 | 是否需要在Onboarding阶段收集用户的信仰阶段信息用于后续推荐？ | P1 | 产品经理 |
| Q02-5 | 产品内是否有慕道友适用的内容？如果有，在Onboarding中如何呈现？ | P2 | 内容团队 |

### ET-03 每次启动App的体验

**核心问题**：用户打开App后，如何帮助用户最快实现当前意图或续接上次未完成的任务？

#### 用户场景

| 场景 | 用户画像 | 触发情境 | 目标 |
|------|---------|---------|------|
| **带着新意图** | | | |
| S3-1 | 想读今天经文的弟兄姐妹 | 清晨灵修时间 | 快速开始今天的读经计划 |
| S3-2 | 想祷告的弟兄姐妹 | 午间休息、睡前 | 快速开始祷告，打开祷告清单 |
| S3-3 | 想查找特定经文的用户 | 查经、聚会中 | 快速找到约3:16或其他特定经文 |
| S3-4 | 想看小组动态的用户 | 闲暇时刻 | 看看小组成员的最新感想和动态 |
| S3-5 | 想记录灵感或感动的用户 | 聚会中、灵修中 | 快速打开笔记开始写 |
| S3-6 | 想找资源（讲道/灵修材料）的用户 | 想深入学习某个主题 | 发现适合自己的内容 |
| **续接上次任务** | | | |
| S3-7 | 上次读到一半的用户 | 通勤途中继续阅读 | 回到上次阅读的位置 |
| S3-8 | 上次听到一半的用户 | 走路/做家务时 | 继续播放上次的有声圣经 |
| S3-9 | 上次写到一半的笔记/感想 | 有时间了继续写 | 恢复草稿，继续编辑 |
| S3-10 | 上次查看的注释/辞典 | 想继续研究 | 找到上次查看的参考材料 |

#### 当前产品能力

基于 `group-product-current-state.md` 的信息：

- **底部导航**：4个Tab（圣经、计划、资源、我的），启动后默认进入圣经Tab
- **圣经Tab首页**：每日灵修卡片 + 功能入口列表（圣经目录、搜索、阅读、有声圣经、灵粮、注释、辞典）
- **计划Tab**：我的计划列表（包含个人、共读、小组计划）
- **资源Tab**：音频灵修资源
- **"我的"Tab**：个人中心、小组入口

**已有的续接能力**：
- 圣经阅读"显示最近阅读位置"（据current-state描述）
- 计划Tab显示"我的计划"列表，可以看到进行中的计划

**缺失的续接能力**：
- 无全局的"上次进行到哪里"提示
- 无跨模块的"最近操作"汇聚视图
- 祷告模块的Daily Tab有Activities混合列表，但与圣经Tab是分离的

#### 体验断点

| 编号 | 断点描述 | 严重程度 | 说明 |
|------|---------|---------|------|
| ET03-G1 | 启动页（圣经Tab）是静态功能菜单而非动态任务引导 | P1 | 首页应该是"你今天可以做什么"的动态推荐，而不是一成不变的功能入口列表 |
| ET03-G2 | 缺少跨模块的"续接上次"能力 | P1 | 用户上次在祷告模块中途退出，重新打开App后进入的是圣经Tab，需要自己导航回去 |
| ET03-G3 | 不同意图的用户看到同一个首页 | P2 | 习惯早晨读经的用户和习惯晚上祷告的用户，首页应该呈现不同内容 |
| ET03-G4 | 推送通知点击后的落地体验 | ⚠️ 待确认 | 点击"今日读经提醒"后是否直达今日任务，还是进入App首页需要自己找 |

#### AI时代的挑战与机会

- **挑战**：用户可能更习惯"对AI说一句话就开始"（"帮我开始今天的灵修"），相比之下多次点击导航显得笨拙

#### AI时代的挑战与机会

- **挑战**：用户可以直接问AI"以弗所书2:8-9是什么意思？"并得到看似合理的解释。AI在语言解释层面已经相当好用

- **机会**：产品内整合的AI辅助解经，可以有以下差异化优势：

  2. **有神学边界**：AI的回应在产品内可以被约束在正统神学范围内，不像通用AI可能给出非正统甚至错误的解释
  3. **有上下文**：AI知道用户正在读什么、之前读过什么、在哪个小组讨论，可以提供更贴合的解释
- **可能的产品形态**：在阅读经文时，提供"问AI"入口，但AI的回答是基于产品内的注释库和可信来源生成的摘要，而非通用训练数据。每个回答附带来源链接

- **挑战**：AI可以生成"祷告词"，用户可能依赖AI写祷告而非发自内心
- **风险**：祷告的核心是与神的个人关系，过度工具化会损害其属灵本质
- **机会**：AI可以在不替代祷告本身的前提下提供辅助——比如基于用户当天的读经内容推荐祷告方向，或者基于祷告历史发现"你最近祷告中经常出现焦虑主题，要不要尝试这段关于平安的经文祷告？"
- **底线**：产品绝不应该提供"AI替你祷告"的功能。可以提供引导和结构，但祷告的内容必须来自用户自己

### ET-08 使用资源

**核心问题**：如何让用户在对的时机遇见对的信仰内容——既达到通用产品培养的内容发现期待，又尊重信仰内容"形塑生命而非消费娱乐"的本质？

> **分析框架说明**：用户对"内容发现"的体验期待是由Spotify、Netflix、YouTube、小红书、喜马拉雅等日常产品塑造的。但信仰内容有根本性的不同——它的目的是**formation（生命塑造）**而非**consumption（内容消费）**。这意味着我们不能简单移植通用产品的发现和推送逻辑，而需要在"让内容可被找到"和"让内容在合适的时机以合适的方式出现"之间找到信仰产品独有的平衡。本节先梳理内容体验的完整生命周期，再针对每个阶段分析"通用标准"和"信仰约束"的张力。

---

#### 8.1 信仰内容发现的独特约束

在进入生命周期分析之前，先明确信仰内容与通用内容的根本差异——这些差异决定了我们不能照搬通用产品的做法：

| 维度 | 通用内容平台（Spotify/Netflix/小红书） | 信仰内容产品 | 对设计的影响 |
|------|--------------------------------------|------------|------------|
| **成功指标** | 更多点击、更长停留、更高频次 | 生命成长、理解深化、信仰实践 | 不能用"engagement时长"衡量成功；用户听完一段讲道去服侍人，比继续刷下一段更好 |
| **内容权威性** | 所有内容平等，由用户偏好决定价值 | 内容有神学准确性要求，不同作者/机构的权威性不同 | 推荐算法不能仅基于点击率——一篇有争议的文章可能点击高但不适合推荐 |
| **推荐逻辑** | "你可能喜欢"（偏好驱动） | "你可能需要"（成长驱动）+ "值得信赖的人推荐"（关系驱动） | 牧者推荐、小组长推荐的权重应高于算法推荐 |
| **内容分级** | 年龄分级（PG-13等） | 信仰阶段适配——初信者、成熟信徒、服侍者需要不同深度的内容 | 需要"信仰阶段"维度的内容标记，而非仅按"热度"排序 |
| **隐私敏感度** | 浏览历史是商业数据 | 浏览历史反映属灵状态——用户正在查看"如何面对怀疑"或"婚姻危机"的内容极为私密 | 浏览记录和推荐结果的可见性需要严格控制；不能在小组中显示"某某最近在看离婚相关文章" |
| **时间关系** | 即时满足，追求新鲜 | 经典内容价值不因时间递减——2000年前的经文和100年前的神学著作依然核心 | 不能像新闻App一样偏重"最新"排序；需要"经典"与"当代"并重 |
| **被动vs主动** | 鼓励被动消费（自动播放、无限滚动） | 鼓励主动寻求和深度停留 | 不应设计"无限滚动的内容流"；应设计"帮助用户找到并深入一个内容"的体验 |
| **社交推荐** | 大众评分、"10万人都在看" | 信任圈推荐——"我的牧师推荐"、"我的小组在学习" | 社交证据的来源应是"属灵信任圈"而非"大众热度" |

#### 8.5 体验断点（按生命周期阶段组织）

##### 阶段一：需求产生 — "我不知道这里有我需要的东西"

| 编号 | 断点描述 | 严重程度 | 说明 |
|------|---------|---------|------|
| ET08-G1 | 资源在App中的存在感极低 | P0 | 资源Tab虽然是一级入口，但用户可能从不点击——因为不知道里面有什么、对自己有什么用。圣经Tab和计划Tab有明确的使用理由（"读圣经"、"我的计划"），资源Tab缺少类似的"为什么要来这里"的理由。**通用标准**：Netflix首页直接展示"继续观看"和个性化推荐，用户不需要主动寻找。**信仰约束**：但我们不应该像Netflix那样用算法"喂食"——应该是"邀请"而非"推送" |
| ET08-G2 | 资源与用户当前的灵修行为完全脱节 | P0 | 用户正在读约翰福音3章，但没有任何地方告诉他"这里有一篇关于约翰福音3:16的精彩讲解"。资源是孤立的，不嵌入在用户的灵修流程中。**通用标准**：Amazon在你浏览商品时推荐"经常一起购买"。**信仰适配**：在经文阅读页面，以不干扰阅读的方式提供"相关资源"入口（如页面底部或侧栏），让用户在需要更深理解时自然发现 |

##### 阶段二：发现内容 — "有宝藏但找不到"

| 编号 | 断点描述 | 严重程度 | 说明 |
|------|---------|---------|------|
| ET08-G3 | 资源不可搜索 | P0 | 大量内容已上线但用户无法搜索。想找"关于饶恕的讲道"没有途径。**通用标准**：任何内容平台的最低期待——输入关键词、得到结果。**信仰适配**：搜索结果应支持按"经文关联"、"内容类型"、"作者/机构"筛选，而不仅是关键词匹配 |
| ET08-G4 | 缺少主题索引/分类体系 | P0 | 用户不能按主题浏览内容。**通用标准**：Spotify有"心情"、"场景"、"流派"等多维度分类。**信仰适配**：分类维度应包括——(a) 神学主题（信心/恩典/圣灵/末世）；(b) 生活处境（婚姻/职场/亲子/悲伤/焦虑）；(c) 灵修类型（每日灵修/深度查经/讲道/见证/诗歌）；(d) 内容形式（文章/音频/视频/课程）；(e) 深度层级（入门/进阶/深度研究）。其中(b)生活处境分类尤为重要——用户遇到困难时的搜索方式是"面对焦虑怎么办"而不是"论神的平安在基督徒生活中的角色" |
| ET08-G5 | 缺少"信任关系"驱动的内容发现路径 | P1 | 当前没有"我的牧师推荐"、"我的小组在学习"、"我的教会常用"等入口。**通用标准**：小红书的"关注"Tab、Spotify的"好友在听"。**信仰适配**：这一点信仰产品应该做得比通用产品更好——因为信仰社群中的信任关系比算法推荐更有价值。小组长为下次查经推荐的一篇文章，比算法推荐的十篇文章更有效 |

##### 阶段三：评估决策 — "这个内容值不值得我花时间？"

| 编号 | 断点描述 | 严重程度 | 说明 |
|------|---------|---------|------|
| ET08-G6 | 缺少帮助用户判断内容价值的信号 | P1 | 用户看到一个资源标题，不知道它多长、多深、是否适合自己。**通用标准**：Netflix有预告片、时长、评分、类型标签；YouTube有缩略图、时长、观看数。**信仰适配**：信仰内容的质量信号不应以"热度"（播放量/点赞数）为主——一个5万播放的鸡汤视频不一定比500播放的深度查经更值得推荐。更合适的信号包括：(a) 内容来源的权威性（知名牧者/神学院/出版机构）；(b) 内容摘要或导读（"本文从三个角度解读饶恕的含义"）；(c) 预估阅读/收听时长；(d) 相关经文列表；(e) 适合的读者（"适合想深入了解恩典教义的信徒"）。而"多少人收藏"可以作为辅助信号——它反映的是"有多少人觉得值得反复阅读"，比"多少人点击过"更有意义 |
| ET08-G7 | 缺少内容预览机制 | P1 | 无法在不进入内容页面的情况下快速判断是否值得深入。**通用标准**：Netflix的悬停预览、Spotify的30秒试听、小红书的封面+标题。**信仰适配**：音频资源提供30秒试听；文章资源提供前3段预览或AI生成的内容摘要；视频提供关键帧预览 |

##### 阶段四：消费体验 — "我开始阅读/收听了"

| 编号 | 断点描述 | 严重程度 | 说明 |
|------|---------|---------|------|
| ET08-G8 | 阅读/收听体验与内容类型不匹配 | P1 | 不同类型的内容需要不同的消费体验——短篇灵修文章需要沉浸式阅读（Reader Mode）；长篇讲道音频需要后台播放和倍速控制（连接ET-09）；系列课程需要进度追踪和章节导航；视频讲解需要全屏播放和笔记并行。当前是否针对不同内容类型提供了适配的消费体验？ |
| ET08-G9 | 资源消费中缺少"划线/笔记/书签"能力 | P1 | 阅读一篇好文章时想标记重点段落、写旁注——当前是否支持？这与ET-07（记笔记）直接相关。**信仰适配**：信仰内容的消费不是"看完就完"，而是"读、思考、记录、应用"的循环——产品应该支持这个循环 |

##### 阶段五：消化吸收 — "我看完了，然后呢？"

| 编号 | 断点描述 | 严重程度 | 说明 |
|------|---------|---------|------|
| ET08-G10 | 看完/听完后缺乏引导和连接 | P2 | 阅读完一篇关于饶恕的文章后，没有引导到下一步——"继续阅读同主题的其他文章"、"将这个主题带入今天的祷告"、"分享给你的小组讨论"。**通用标准**：YouTube的"推荐下一个"、Netflix的"你可能还喜欢"。**信仰约束**：不应该像YouTube那样用"自动播放下一个"来延长停留时间。更合适的做法是：完成后显示2-3个有意义的"下一步"选项——可以是继续深入（同主题更深的内容）、横向扩展（相关主题）、或实践应用（将所学带入祷告/笔记/小组讨论）。让用户选择，而不是自动推进 |

##### 阶段六：收藏管理 — "好内容我想留下来反复看"

| 编号 | 断点描述 | 严重程度 | 说明 |
|------|---------|---------|------|
| ET08-G11 | 缺少收藏/保存机制 | P1 | 遇到好内容想保存以后再看——需要"收藏"或"书架"功能。**通用标准**：YouTube的"Watch Later"、微信的"收藏"、Safari的"阅读列表"。**信仰适配**：收藏应支持分类整理（"查经材料"、"灵修文章"、"讲道合集"），方便反复使用，尤其是小组长需要管理自己积累的查经资源 |
| ET08-G12 | 缺少阅读/收听历史 | P1 | "上周看过一篇很好的文章想再找到"——需要浏览历史。与ET-09的播放历史是同一个问题在不同内容类型上的体现 |

##### 阶段七：分享推荐 — "好东西想推荐给别人"

| 编号 | 断点描述 | 严重程度 | 说明 |
|------|---------|---------|------|
| ET08-G13 | 资源无法分享到小组 | P1 | 小组长找到一篇好的查经材料，想推荐给组员——当前无法在小组内分享资源链接。**信仰适配**：这是信仰产品最应该做好的分享——不是"我觉得好玩你也看看"的娱乐分享，而是"这篇文章对我们下次查经有帮助"的属灵建造分享。分享时应允许附加推荐理由（"推荐大家在本周聚会前阅读"） |
| ET08-G14 | 资源无法分享到外部（微信等） | P2 | 想把好内容分享到微信群/朋友圈——需要生成分享卡片。**通用标准**：小红书的分享图片、Spotify的分享链接。**信仰适配**：分享卡片的设计应温暖而非推销——"一位弟兄分享了这篇灵修文章"而非"快来看10万人都在读的热门文章" |

##### 跨阶段问题：资源的产品定位

| 编号 | 断点描述 | 严重程度 | 说明 |
|------|---------|---------|------|
| ET08-G15 | 资源模块在App中的角色定位不清 | P0（战略级） | 资源Tab是"独立的内容库"（用户主动来浏览）还是"辅助其他模块的工具"（经文阅读时推荐相关资源）？还是两者兼有？这个定位决定了信息架构和资源投入的方向。**需要产品团队做出明确决策** |

#### 8.7 信仰产品的内容发现分层框架

基于通用标准和信仰约束的平衡，提出分层框架：

```
┌─────────────────────────────────────────────────────────────┐
│  Layer 4: 关系驱动发现（信仰产品的独特优势）                  │
│  • 牧者/小组长推荐入口——"你的牧师推荐了这篇文章"              │
│  • 小组查经材料推荐——"你的小组下周要学习的内容"                │
│  • 教会定制内容列表——"恩典教会推荐的灵修材料"                 │
│  • 弟兄姐妹的分享——"李弟兄分享了一篇关于信心的见证"           │
├─────────────────────────────────────────────────────────────┤
│  Layer 3: 上下文关联发现（让资源嵌入灵修流程）                │
│  • 经文→相关资源（"读约翰福音3章时，推荐这篇讲解"）           │
│  • 祷告主题→相关资源（"你在为婚姻祷告，这篇文章可能帮助你"）  │
│  • 计划→配套资源（"本周计划经文的背景知识"）                  │
│  • 处境→相关资源（"面对焦虑时的圣经安慰"）                   │
├─────────────────────────────────────────────────────────────┤
│  Layer 2: 主动探索发现（用户有意识地寻找）                    │
│  • 搜索（关键词+经文+主题+作者）                              │
│  • 分类浏览（按神学主题/生活处境/内容类型/深度层级）           │
│  • 编辑策展（"本月推荐"、"查经入门必读"）                     │
│  • 收藏管理（我保存的内容、我的学习记录）                     │
├─────────────────────────────────────────────────────────────┤
│  Layer 1: 基础可访问（及格线）                               │
│  • 资源Tab有清晰的内容组织（而非一堆列表）                    │
│  • 内容元数据完整（标题、作者、时长、类型、摘要）              │
│  • 首页有资源的存在感（"继续阅读"、"今日灵修推荐"）            │
│  • 内容阅读/收听体验达到基本可用                              │
└─────────────────────────────────────────────────────────────┘
```

**关键洞察**：

- **Layer 4是信仰产品的真正护城河**。通用产品依赖算法推荐，信仰产品可以依赖**关系推荐**——这不是退而求其次，而是更高质量的推荐。一个牧者了解会众的属灵状态和需要，他的推荐精准度远超算法。
- **Layer 3是实际体验的关键**。大多数用户不会主动去"逛"资源Tab。如果资源只存在于资源Tab中，它就永远是一个孤岛。资源必须出现在用户正在做的事情的上下文中——读经时、祷告时、参与小组时。
- **Layer 2是保底的通用能力**。搜索和分类是"有就不会被夸，没有会被骂"的基础设施。
- **Layer 1是当前最紧急的**。基础都没有，上层建筑无从谈起。

#### 8.8 关于"算法推荐"的信仰产品立场

> 这是一个需要产品团队明确立场的战略问题。

通用产品的推荐算法基于一个假设：**用户的行为数据反映了他们想要什么，而给他们更多他们想要的就是好的**。这在娱乐领域是合理的，但在信仰领域可能有问题：

- 一个正在经历信仰危机的用户频繁搜索"怀疑"相关内容——算法会推荐更多关于怀疑的内容，但牧者可能会推荐关于"信心的坚固"的内容
- 一个用户只看浅层灵修文章——算法会继续推荐浅层内容（因为点击率高），但他可能需要被引导向更深层的内容
- "热门"推荐可能把流量集中在少数内容上，而一些小众但高质量的深度内容永远得不到曝光

**建议立场**：信仰产品应优先采用**"策展+关系"推荐**而非**"算法"推荐**：

| 推荐方式 | 优先级 | 说明 |
|---------|--------|------|
| 编辑策展 | 最高 | 由内容团队/神学顾问精选的推荐——"本月灵修精选"、"查经入门五篇必读" |
| 关系推荐 | 高 | 牧者推荐、小组长推荐、教会推荐——基于属灵信任关系 |
| 上下文关联 | 高 | 经文→资源、主题→资源——基于内容的客观关联，不是用户行为 |
| 行为辅助 | 辅助 | "你最近在读约翰福音，这里有一个相关的灵修系列"——基于用户行为但限于**当前灵修上下文**，不做跨域推荐 |
| 热度排序 | 最低 | 仅在用户主动浏览列表页面时作为排序选项之一，不作为默认 |

#### 8.9 AI时代的挑战与机会

- **挑战**：用户可以直接问ChatGPT"约翰福音3:16怎么理解？"、"面对焦虑时圣经怎么说？"——比在资源库里翻找文章效率高得多
- **机会**：
  1. **AI作为连接器而非替代者**：用户问AI问题时，AI不仅回答，还推荐App内经过审核的高质量资源——"关于这个话题，你可能对这篇文章感兴趣"。AI回答是即时但浅层的，App内资源是深度但需要时间的——两者互补
  2. **AI驱动的内容摘要和导读**：为每篇资源生成简短摘要，帮用户快速判断是否值得深入——这解决ET08-G7（预览机制）的问题
  3. **AI辅助的经文-资源关联**：自动分析资源内容，建立资源与经文的关联图谱——这大幅降低内容标注的人工成本
  4. **AI辅助搜索**：用户用自然语言描述需求（"关于如何在职场活出信仰的内容"），AI理解意图并找到相关资源——比关键词搜索更好
- **底线**：AI生成的内容摘要和推荐理由应标注为"AI辅助生成"，且不替代编辑策展的优先级地位

### ET-09 视听体验

**核心问题**：如何提供达到用户日常使用的通用媒体App（Spotify、YouTube、播客App）同等水准的音视频体验，同时利用信仰内容的独特属性创造差异化价值？

> **分析框架说明**：用户对音视频体验的期待不是由同类圣经App塑造的，而是由Spotify、YouTube、Apple Podcasts、Netflix、喜马拉雅等日常使用的通用产品塑造的。本节按音视频体验的完整生命周期进行系统梳理，而非仅列举"有什么功能"。

#### 音视频体验生命周期

```
发现内容 → 开始播放 → 播放控制 → 中断/恢复 → 收藏/管理 → 场景适配 → 社交分享
  (1)        (2)        (3)        (4)         (5)        (6)        (7)
```

#### 9.1 用户场景（按使用情境分类）

**注意力程度**是理解音视频场景的关键维度——它决定了界面设计和交互模式：

| 场景 | 注意力 | 触发情境 | 典型内容 | 关键体验要求 |
|------|--------|---------|---------|------------|
| **专注灵修** | 高 | 清晨/睡前安静时间 | 灵修音频、祷告引导音乐 | 沉浸感、无干扰、自动连续播放 |
| **深度学习** | 高 | 查经准备、系统学习 | 讲道视频、圣经讲解系列 | 视频+笔记并行、书签标记、章节跳转 |
| **读听同步** | 高 | 个人灵修时间 | 有声圣经+文本 | 文本高亮跟随音频位置、随时切换模式 |
| **通勤收听** | 低 | 开车/地铁/步行 | 有声圣经、讲道音频、灵修播客 | 后台播放、锁屏控制、断点续播、大按钮 |
| **家务伴听** | 低 | 做饭/打扫/运动 | 同上 | 语音控制（理想情况）、定时关闭、连续播放 |
| **车载播放** | 极低 | 开车 | 有声圣经、讲道 | CarPlay/Android Auto适配、极简界面、语音操控 |
| **入睡伴听** | 递减 | 睡前 | 诗篇朗读、轻音乐、灵修冥想 | 定时关闭（Sleep Timer）、音量渐弱、界面暗色 |
| **碎片浏览** | 中 | 等待间隙 | 短视频讲解、经文朗读片段 | 快速加载、迷你播放器、不打断当前浏览 |

#### 9.3 体验断点（按生命周期阶段组织）

##### 阶段一：发现内容

| 编号 | 断点描述 | 严重程度 | 说明 |
|------|---------|---------|------|
| ET09-G1 | 音视频内容缺乏独立的发现入口 | P1 | 音频和视频散布在不同模块（圣经Tab的有声圣经、资源Tab的音频灵修、祷告模块的音乐），没有统一的"听/看"入口。用户不知道App里到底有多少可听可看的内容。对比：Spotify有清晰的"音乐"和"播客"分类；喜马拉雅有"发现"页 |
| ET09-G2 | 音视频内容不可搜索 | P1 | 与ET-08的资源搜索问题相同——想找"关于饶恕的讲道"没有搜索入口 |

##### 阶段二：播放核心

| 编号      | 断点描述          | 严重程度   | 说明                                                                                                   |
| ------- | ------------- | ------ | ---------------------------------------------------------------------------------------------------- |
| ET09-G3 | 播放器基础功能完整性    | ⚠️ 待确认 | 用户对播放器的**最低期待**（由日常App塑造）包括：进度条拖动、倍速（0.5x~2.0x）、快进/快退15s或30s、音量控制。这些是及格线，不是加分项。对比：任何主流播客App都支持全部这些   |
| ET09-G4 | 迷你播放器         | ⚠️ 待确认 | 播放音频时浏览App其他页面（如查看小组、翻阅圣经），底部应有迷你播放器持续显示当前播放状态和暂停/播放控制。对比：Spotify、Apple Music、YouTube Music的迷你播放器是标配 |
| ET09-G5 | 后台播放和锁屏控制     | ⚠️ 待确认 | 切到其他App或锁屏后音频是否继续播放？是否出现系统级锁屏控制面板（封面、标题、进度条、播放/暂停/上下曲）？如果不支持，音频功能在通勤和家务场景下基本不可用。对比：所有主流音频App的基础能力    |
| ET09-G6 | 跨内容类型的播放体验不一致 | P2     | 有声圣经、灵修音频、讲道视频、祷告背景音乐——这些可能使用不同的播放器组件，导致体验不一致（进度条样式不同、功能按钮位置不同、倍速支持程度不同）。对比：Spotify对音乐和播客使用统一播放器框架   |

##### 阶段三：连续性与播放记录

| 编号 | 断点描述 | 严重程度 | 说明 |
|------|---------|---------|------|
| ET09-G7 | **断点续播**——用户没有听完/看完时，下次回来能否从上次位置继续 | P0 | 这是你指出的核心问题。用户极少一次性听完整卷有声圣经或整集讲道视频。如果每次重新打开都从头开始，体验极差。需要为**每个内容单独记住播放位置**。对比：Audible的Whispersync记住每本书精确到秒的位置；YouTube记住每个视频的观看进度（进度条上显示红色已观看标记）；Netflix在封面上直接显示"观看至 45:23" |
| ET09-G8 | **播放历史记录**——用户无法查看和回溯最近听过/看过的内容 | P1 | 用户可能想："上周听的那个讲道很好，想再听一遍"或"昨天听到一半的灵修，今天继续"。没有播放历史就无法找回。对比：YouTube的"历史记录"、Spotify的"最近播放"、Apple Podcasts的"最近播放"——这是所有媒体App的标配 |
| ET09-G9 | **跨设备同步播放进度** | P2 | 手机上听到一半，换到平板上继续——需要进度同步。对比：Audible的Whispersync、Spotify的跨设备续播 |

##### 阶段四：队列与收藏管理

| 编号 | 断点描述 | 严重程度 | 说明 |
|------|---------|---------|------|
| ET09-G10 | **播放队列（Queue）**——用户无法排列接下来要听的内容 | P1 | 用户可能想："先听完这章以赛亚书，接着听那篇灵修，然后听创世纪下一章"。没有队列就必须手动一个个找。对比：Spotify的播放队列（"Next in Queue"）允许用户随时添加、排序、移除；Apple Music的"接下来播放"。最基础的版本是"自动播放下一个"（如有声圣经听完第3章自动播第4章） |
| ET09-G11 | **播放列表（Playlist）**——用户无法创建和管理自己的音视频收藏 | P2 | 与队列不同，播放列表是持久化的收藏。用户可能想创建"我的灵修音乐"、"约翰福音讲道系列"、"睡前诗篇朗读"等个性化列表。对比：Spotify的播放列表、YouTube的"播放清单"。需要评估：在我们的产品中，这个需求有多强？还是"收藏/稍后再听"就够了？ |
| ET09-G12 | **"稍后再听/看"功能** | P1 | 发现了一个好内容但现在没时间——需要一个"稍后再听"的收纳入口。比完整的播放列表更轻量，但同样重要。对比：YouTube的"Watch Later"、Safari的"阅读列表"、播客App的"待播放" |
| ET09-G13 | **自动连续播放** | P1 | 有声圣经听完第3章是否自动播第4章？灵修系列听完第1集是否自动播第2集？如果不能，低注意力场景（通勤、家务）下用户必须频繁拿起手机操作。对比：YouTube的自动播放下一个视频、Spotify的自动播放、播客App的连续播放 |

##### 阶段五：场景适配

| 编号 | 断点描述 | 严重程度 | 说明 |
|------|---------|---------|------|
| ET09-G14 | **定时关闭（Sleep Timer）** | P1 | 睡前听诗篇朗读或祷告音乐的用户需要设置"30分钟后停止播放"或"本章结束后停止"。没有这个功能，要么手机一直播到天亮，要么用户必须保持清醒去手动暂停。对比：Audible（精确到分钟+章节结束两种模式）、Headspace（冥想结束自动停止）、所有主流播客App |
| ET09-G15 | **车载体验** | P2 | 开车时操作手机是安全隐患。需要考虑：(1) CarPlay / Android Auto 适配——在车载屏幕上显示极简播放界面；(2) 蓝牙连接后自动恢复上次播放；(3) 方向盘控制键的响应（下一曲/上一曲映射为下一章/上一章）。对比：Spotify的车载模式（大按钮、极简界面）、Apple Maps的音频控制集成 |
| ET09-G16 | **低注意力场景的交互适配** | P2 | 通勤、运动、家务时操作应该是"盲操作友好"的——大按钮、简单手势（如摇一摇切换）、语音控制（理想状态）。当前播放器如果只有小按钮和精确拖动进度条，在这些场景下不可用。对比：Overcast的"Smart Speed"和大面积点击区域 |

##### 阶段六：文本-音频同步（信仰App特有）

| 编号 | 断点描述 | 严重程度 | 说明 |
|------|---------|---------|------|
| ET09-G17 | **读经与听经的位置同步** | P1 | 听有声圣经时，文本页面应同步高亮当前朗读位置（逐节或逐段）；反过来，从文本页面切换到听经模式时，应该从当前阅读位置开始播放。这是**圣经App独有的差异化体验**——通用音频App做不到。对比：Kindle的Whispersync for Voice（有声书和电子书位置同步）、YouVersion的音频文本同步滚动 |
| ET09-G18 | **阅读/聆听模式的自由切换** | P2 | 灵修时可能想先听一遍经文，再切到文本仔细研读某节，然后再切回听下一段。这种高频切换应该是零摩擦的——一键切换，不丢失位置 |

##### 阶段七：社交与分享

| 编号 | 断点描述 | 严重程度 | 说明 |
|------|---------|---------|------|
| ET09-G19 | **分享音视频内容到小组或外部** | P2 | 听到一个好的讲道想分享给小组——需要"分享"入口（连接ET-10）。分享到微信时应该生成精美的卡片（标题+封面+时长+App链接）。对比：喜马拉雅的分享卡片、Spotify的分享 |
| ET09-G20 | **小组共同收听** | P3 | 小组查经时组长播放一段讲解，大家一起听——是否有"小组广播"的需求？这是一个需要验证的场景 |

#### 9.5 信仰App的音视频体验分层框架

基于竞品分析和用户场景，提出分层框架，帮助确定优先级：

```
┌─────────────────────────────────────────────────────────────┐
│  Layer 4: 差异化体验（信仰App独有价值）                       │
│  • 文本-音频位置同步（读经↔听经无缝切换）                     │
│  • 祷告引导音频+背景音乐的沉浸体验                            │
│  • 经文时间戳书签/评论                                       │
│  • 信仰音乐歌单的社群分享                                    │
├─────────────────────────────────────────────────────────────┤
│  Layer 3: 高级体验（提升满意度和粘性）                        │
│  • 播放队列（Queue）和自动连续播放                             │
│  • 播放列表/歌单（用户自建）                                  │
│  • "稍后再听"收藏                                            │
│  • CarPlay / Android Auto 适配                               │
│  • 定时关闭（Sleep Timer）                                    │
│  • 离线下载管理                                              │
│  • 倍速播放                                                  │
├─────────────────────────────────────────────────────────────┤
│  Layer 2: 连续性体验（留住用户的关键）                        │
│  • 断点续播（记住每个内容的播放位置）                          │
│  • 播放历史记录                                              │
│  • 首页"继续收听/收看"入口                                    │
│  • 跨设备进度同步                                            │
├─────────────────────────────────────────────────────────────┤
│  Layer 1: 播放基础（及格线，没有就不能用）                     │
│  • 播放/暂停/进度条拖动                                       │
│  • 后台播放 + 锁屏控制面板                                    │
│  • 迷你播放器（浏览其他页面时不中断播放）                      │
│  • 基本的前进/后退跳转（15s/30s）                             │
└─────────────────────────────────────────────────────────────┘
```

**关键洞察**：Layer 1和Layer 2是**用户不会赞赏但没有会抱怨**的能力。如果连后台播放和断点续播都没有，用户会认为这不是一个"正经的"音频产品。Layer 3和Layer 4才是用户愿意"为之选择这个App"的原因。

#### 9.6 AI时代的挑战与机会

- **挑战**：AI语音合成质量快速提升（如ElevenLabs、Azure Neural TTS），用户可以用AI朗读任何文本——包括圣经经文、注释、灵修文章
- **机会**：
  1. **TTS扩展内容范围**：将非音频内容（注释、灵修文章、用户笔记）用高质量TTS转为可听内容，大幅扩展"可听"的内容池。这让用户不再只能听"有声圣经"和少量已录制的灵修音频，而是**App内的一切文字内容都可以听**
  2. **智能字幕和转录**：为视频内容自动生成字幕和文字稿，使视频内容可搜索、可引用、可做笔记
  3. **个性化音频灵修**：基于用户当天读经内容，AI合成个性化灵修引导音频（需谨慎处理神学准确性——仅用于朗读已审核的文本，不生成新内容）
  4. **音频内容的智能摘要**：听完一段30分钟讲道后，AI生成要点摘要+关键经文列表，帮助用户回顾和记笔记
- **底线**：AI辅助的是内容的"可听化"和"可搜索化"，不应替代专业录制的有声圣经朗读体验（专业朗读的情感表达、断句韵律是TTS尚无法完全替代的）

#### 9.9 优先级建议

基于分层框架，建议的实施顺序：

```
Phase 1 — 确保及格线（Layer 1）
├── 验证并补齐：后台播放、锁屏控制、迷你播放器
├── 验证并补齐：基本播放控制（倍速、快进快退、进度拖动）
└── 预估：如果基础已有，验证即可；如果缺失，这是P0级开发任务

Phase 2 — 建立连续性（Layer 2）
├── 实现断点续播（每个内容记住播放位置）
├── 实现播放历史记录
├── 首页增加"继续收听"入口（连接ET-03首页改造）
└── 预估：这是用户从"偶尔用"变为"每天用"音频功能的关键

Phase 3 — 提升体验（Layer 3）
├── 自动连续播放（听完一章自动播下一章）
├── Sleep Timer
├── 播放队列（Queue）
├── "稍后再听"收藏
├── 倍速播放优化（更多速度选项）
├── 离线下载管理
└── 预估：让音频功能从"能用"变为"好用"

Phase 4 — 差异化（Layer 4）
├── 文本-音频位置同步（读经↔听经）
├── 车载模式 / CarPlay / Android Auto
├── 播放列表（用户自建）
├── TTS扩展内容（注释/文章可听化）
└── 预估：这是信仰App独有的竞争优势，但依赖Phase 1-3的基础
```

### ET-11 习惯养成

**核心问题**：如何帮助用户将一次性的使用变为持续的属灵操练习惯？

#### 用户场景

| 场景 | 用户画像 | 触发情境 | 目标 |
|------|---------|---------|------|
| S11-1 | 刚开始读经计划的新用户 | 第一周 | 养成"每天打开App读经"的习惯 |
| S11-2 | 坚持了30天的用户 | 中途想放弃 | 看到自己的进步和成就，保持动力 |
| S11-3 | 中断后想恢复的用户 | 出差/生病导致中断一周 | 不因为中断而产生"全部白费"的挫败感 |
| S11-4 | 想建立多种操练的用户 | 不只是读经，还想祷告、灵修 | 管理多个属灵操练习惯 |
| S11-5 | 小组成员互相激励 | 看到别人在坚持 | 社群压力和鼓励帮助自己坚持 |

#### 当前产品能力

- **读经计划**：提供结构化的读经进度，有每日任务和完成勾选
- **计划进度可视化**：待确认具体呈现形式
- **提醒推送**：读经提醒（已列为P0待开发，参见 `group-experience-gaps.md § 2.5`）

**缺失**：
- 跨功能的习惯追踪（不仅追踪读经，也追踪祷告、灵修）
- 连续性激励（连续打卡记录、里程碑庆祝）
- 中断恢复的优雅处理
- 个性化的习惯推荐

#### 体验断点

| 编号 | 断点描述 | 严重程度 | 说明 |
|------|---------|---------|------|
| ET11-G1 | 缺少每日提醒推送 | P0 | 已在 `group-experience-gaps.md § 2.5` 记录——没有外部触发，依赖用户自觉 |
| ET11-G2 | 缺少连续性激励机制 | P1 | 没有"连续第7天""连续第30天"等里程碑庆祝和可视化 |
| ET11-G3 | 中断后的恢复体验不友好 | P1 | 中断几天后打开App，看到"落后了5天"，可能产生焦虑和放弃心理 |
| ET11-G4 | 习惯养成仅限于读经计划 | P2 | 祷告、灵修等其他操练没有类似计划的习惯养成支持 |
| ET11-G5 | 缺少个人属灵操练的整体视图 | P2 | 用户无法在一个地方看到"我的读经、祷告、灵修整体表现如何" |
| ET11-G6 | 游戏化程度需要谨慎平衡 | 设计约束 | 属灵操练不应该过度游戏化（参见 `group-product-vision.md § 4.1`："不做排行榜"），需要在"激励"和"敬虔"之间找平衡 |

#### 竞品参考

| 竞品 | 做法 | 值得借鉴的点 |
|------|------|------------|
| **YouVersion** | 读经连续天数（Streaks）显著展示；完成计划后发放"徽章"；社交排行榜可选开启 | Streaks是最有效的习惯激励机制之一 |
| **Duolingo** | 连续学习火焰图标+冰冻道具（允许偶尔中断不打破连续）；每日目标可调节；周排行榜 | "冰冻"机制很适合信仰操练——安息日/出差可以豁免而不打断连续 |

#### AI时代的挑战与机会

- **挑战**：AI没有"持续性"概念——每次对话都是独立的。习惯养成需要跨时间的追踪和提醒，这是AI的弱项

  1. **智能提醒**：不只是定时提醒，而是基于用户行为模式调整提醒时间和内容
  2. **中断理解**：AI分析中断原因（"你上周出差了？没关系，我帮你调整了计划"）
  3. **个性化激励**：基于用户偏好选择激励方式——有的人喜欢数据可视化，有的人喜欢鼓励性经文

### ET-13 在线/离线连续性（基础设施层）

**核心问题**：如何确保用户在网络不可用时仍能使用核心功能，并在恢复网络后无缝同步？

> 此主题是基础设施层约束，影响所有其他体验主题的设计决策。

#### 离线重要性分析

对圣经App来说，离线能力是**刚需而非nice-to-have**：
- 教会聚会场所（地下室、偏远地区）信号差
- 飞行模式中的灵修需求
- 通勤地铁中的读经/听经
- 国际旅行中无数据漫游

#### 各体验主题的离线需求矩阵

| 体验主题 | 离线必须 | 离线理想 | 仅在线 |
|---------|---------|---------|--------|
| ET-04 找经文 | 圣经文本检索 | 注释查看 | 语义搜索 |
| ET-05 理解圣经 | 已下载的注释 | 辞典 | AI辅助、在线资源 |
| ET-06 祷告 | 祷告清单查看和新增 | 祷告模式（音乐需离线包） | 小组祷告同步 |
| ET-07 记笔记 | 离线编辑和自动保存 | 草稿云同步 | 分享到小组 |
| ET-08 使用资源 | 已下载的资源 | — | 搜索、推荐、在线内容 |
| ET-09 视听 | 已下载的有声圣经 | 已下载的灵修音频 | 在线视频流 |
| ET-10 社交 | — | 离线查看已缓存动态 | 所有互动操作 |
| ET-11 习惯养成 | 今日任务查看和完成标记 | 进度可视化 | 同步和排行 |

#### 关键设计决策

| 决策点 | 选项 | 推荐 |
|--------|------|------|
| 圣经文本是否默认全部离线 | A: 全部下载 B: 按需下载 | A——圣经文本体积不大（约10-20MB/译本），作为核心功能应默认可用 |
| 离线编辑冲突如何处理 | A: 最后写入胜出 B: 提示用户手动合并 | B for 长内容（笔记），A for 简单操作（打卡） |
| 离线状态如何提示用户 | A: 全局提示条 B: 操作失败时提示 C: 功能灰显 | C + B——可离线的功能正常使用，不可用的灰显并在操作时提示 |

#### 待澄清问题

| 编号 | 问题 | 优先级 | 需要谁来回答 |
|------|------|--------|------------|
| Q13-1 | 圣经文本是否已支持全离线？ | P0 | 客户端开发 |
| Q13-2 | 用户本地数据占用情况？（含有声圣经、资源等） | P1 | 客户端开发 |
| Q13-3 | 离线数据的同步策略和冲突处理机制是什么？ | P1 | 后端/客户端 |
| Q13-4 | 用户是否反馈过离线相关问题？频率如何？ | P1 | 客服/用户研究 |

### 矩阵洞察

**1. 圣经模块承载最多体验主题**（7个关联，其中5个核心），说明圣经阅读是整个产品的基石。圣经模块的体验质量直接决定了大多数用户的满意度。

**2. 小组模块涉及面广但深度不足**（7个关联，其中3个核心），验证了已知问题：小组功能被很多体验主题需要，但实际能力不足（使用率<10%）。

**3. 资源模块参与度低**（5个关联，仅2个核心），说明资源目前处于孤立状态。如果能加强资源与圣经/计划/小组的关联，可以大幅提升整体体验。

**4. "我的"Tab缺乏体验主题支撑**，目前只是导航入口，不承载独立的体验价值。可以考虑将其转变为"个人属灵面板"——整合习惯追踪、笔记管理、个人设置等。
