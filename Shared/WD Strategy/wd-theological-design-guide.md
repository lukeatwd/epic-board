## 第一部分：核心神学概念的产品转化

### 1.1 确定性成圣（Gaffin）

**神学核心**：信徒在与基督联合的那一刻，**已经**成为圣洁的人。成圣不是"变成"，而是"活出已是"。

**对齐原则**（Gaffin 确定性成圣）：
- 联合是**已然发生的事实**，不是需要被「创造」的目标
- 产品的角色是**揭示和提醒**，而非**生产和赋予**
- 真正的转化动力来自圣灵，不是来自 App

| 错误的产品描述 | 正确的产品描述 |
|-------------|-------------|
| 「帮助用户成长为更像基督的人」 | 「帮助用户活出已经与基督联合的现实」 |
| 「促进用户的属灵转化」 | 「对齐用户与已然发生的联合」 |
| 「提升用户的灵性水平」 | 「显明用户在基督里的真实身份」 |

#### UI/UX 设计原则

| 原则 | 具体实现 |
| ---- | -------- |
| 身份先于行为 | 任何功能入口先显示身份宣告，再呈现行动选项 |
| 恩典叙事 | 失败场景使用"欢迎回来"而非"你落后了" |
| 已然语态 | 使用"你是..."而非"你要成为..." |

#### 文案示例

| 场景 | 错误设计 | 正确设计 |
| ---- | -------- | -------- |
| 开屏问候 | "今天要完成读经计划" | "你在基督里是新造的人。神的话语今天为你预备好了" |
| 灵修完成 | "做得好！继续努力" | "神的话语已经临到你。这是他给你的礼物" |
| 断更恢复 | "你已经3天没读经了" | "你的身份从未改变。欢迎回家" |
| 读经计划订阅 | "开始你的属灵成长之旅" | "神的话语已为你预备。让我们一起进入" |

#### 触发条件

- **应用场景**：所有与用户成就、进度、状态相关的界面
- **检查点**：产品评审时检查文案是否符合"指示→命令"顺序

### 1.2 十架神学（Gaffin）

**神学核心**：复活生命不是绕过十字架的"胜利后景"，而是"在十字架内被显明"。瓦器必须破裂，宝贝才能流溢。

#### UI/UX 设计原则

| 原则 | 具体实现 |
| ---- | -------- |
| 允许瓦器破裂 | 为失败、挣扎、哀歌提供专门空间 |
| 不强行正面化 | 苦难场景不使用"一切都会好起来的" |
| 苦难是显明 | 苦难中引导用户看见神的工作，而非回避苦难 |

#### Gaffin 十架神学的三个关键洞见

| 洞见 | 传统误解 | Gaffin 的校正 | 对产品的含义 |
|-----|---------|--------------|------------|
| **苦难是「显明」而非「缺席」** | 苦难中神离开了我 | 林后 4:10-11：苦难**显明**基督的生命 | 不说「神与你同在」（感觉不到），而说「苦难正在显明基督的生命在你里面」 |
| **受苦-荣耀是「同时性」而非「序列」** | 先受苦，后得荣耀 | 苦难本身就是荣耀的一部分——瓦器破裂是为了光照出来 | 不承诺「苦难会过去」，而是「苦难中神的工作正在发生」 |
| **不可浪漫化死亡** | 死亡是「回天家」的浪漫旅程 | 死亡是最后仇敌——但基督已经战胜 | 承认死亡的残酷，同时宣告基督的得胜 |

#### 林后 4:7-11 的设计应用（Gaffin 解经）

| 经文 | 神学含义 | 产品应用 |
|-----|---------|---------|
| 「瓦器里的宝贝」 | 有限、脆弱的器皿承载无限的荣耀 | 正常化用户的脆弱——「你是瓦器，这是设计」（Kapic） |
| 「莫大的能力出于神」 | 能力不是来自瓦器，而是来自宝贝 | 不是「你要坚强」，而是「神的能力在你的软弱中显明」 |
| 「身上常带着耶稣的死」 | 苦难是与基督受苦的同参 | 不是「为什么是我」，而是「你正在与基督一同受苦」 |
| 「耶稣的生显明」 | 苦难是显明基督生命的管道 | 苦难不是神离开，而是神的生命正在显明 |

#### 哀歌模式触发条件

| 触发信号 | 检测方式 | 响应行为 |
| -------- | -------- | -------- |
| 关键词检测 | 用户输入含"痛苦""绝望""离开""死亡"等 | 切换为哀歌模式响应 |
| 情绪标记 | 用户主动标记"今天很难" | 推送哀歌诗篇，减少任务提醒 |
| 长期沉默 | 活跃用户突然停用7天以上 | 复归时使用温和问候，不提及"落后" |
| 祷告记录 | 祷告内容含负面情绪关键词 | 主动建议联系牧者/小组长 |

#### 哀歌模式设计原则（整合 Gaffin + Ferguson）

| 原则 | 实施 | 神学根基 |
|-----|-----|---------|
| 同在提醒优于教导 | 减少解释，增加倾听；不急于"解决问题" | Ferguson：「叹息」是圣灵临在的证据 |
| 哀歌经文优先 | 推荐诗篇哀歌、约伯记、耶利米哀歌 | 允许悲伤是圣经的教导 |
| 承认奥秘 | 不给廉价答案；承认苦难的奥秘 | Gaffin：不可浪漫化苦难 |
| 瓦器叙事 | 强调「瓦器破裂是为了光照出来」 | 林后 4:7-11 |
| 引向真实群体 | 主动建议联系牧者、小组长或专业辅导 | Layer 2 教会边界 |
| 安全网 | 检测到自杀/自残信号时，强制显示求助热线 | 关怀生命的责任 |

#### 文案示例

| 用户表达 | 错误响应 | 正确响应 |
| -------- | -------- | -------- |
| "我很痛苦" | "一切都会好起来的！" | "你的痛苦是真实的。神与哀哭的人同哭。这里有一篇诗篇，许多圣徒也曾如此呼求。" |
| "感觉不到神" | "你需要更多祷告" | "感觉的缺失不等于神的缺席。许多圣徒都经历过'属灵的干旱'。你不孤单。" |
| "为什么神不回应" | "神有更好的计划" | "这个问题没有简单答案。我听到你的挣扎。建议你和牧者谈谈。" |
| "我想放弃信仰" | "不要这样想！" | "信仰中的怀疑是真实的。这里有一位牧者的联系方式，他愿意倾听你。" |

#### 旷野场景响应（十架神学校准）

| 用户状态 | ❌ 不当响应（廉价安慰） | ✅ 适切响应（十架神学） | 神学根基 |
|---------|---------------------|---------------------|---------|
| 丧子的父亲 | "神的旨意是美善的" | "我听到你的痛苦。诗篇22篇的作者也曾呼喊'我的神，我的神，为什么离弃我'。你不需要现在就有答案。" | 允许哀歌 |
| 确诊重病 | "凡事谢恩" | "这个消息一定很沉重。林后4:16说'外体虽然毁坏，内心却一天新似一天'——神的工作不止于身体。你愿意一起祷告吗？" | 瓦器叙事 |
| 信仰怀疑 | 立刻辩护神学论证 | "怀疑是信仰旅程的一部分。你愿意告诉我是什么让你产生这些想法吗？" | 同在优先 |
| 长期抑郁 | "多祷告就好了" | "抑郁是真实的挣扎。圣灵在罗8:26说'用说不出来的叹息替我们祷告'——你的叹息本身就是祷告。你是否愿意让我帮你联系一位牧者或辅导？" | Ferguson: 叹息是圣灵工作 |

> **瓦器与宝贝的统一框架**：十架神学最终指向一个自我理解——「我们是瓦器，不是宝贝——但有意识地让自己成为宝贝可以流溢的瓦器」。瓦器的价值不在于自身的完美或技术的先进，而在于它所承载的恩典；瓦器的破裂不是缺陷，而是设计——「要显明这莫大的能力是出于神，不是出于我们」（林后 4:7）。完整论述详见 [wd-ministry-and-technology-theology.md](./wd-ministry-and-technology-theology.md) §14-15。

### 1.3 荣福直观/与神同住（Boersma + Morales）

**神学核心**：人被造的终极目的是"面对面见神"。从伊甸的失落到新耶路撒冷的实现，圣经讲述的是神为人类开辟道路、使人能住在神圣临在中的故事。

#### UI/UX 设计原则

| 原则 | 具体实现 |
| ---- | -------- |
| 目的是相遇 | 灵修定位为"与神相遇"，而非"完成任务" |
| 进入圣所 | 设计"进入"的仪式感（如呼吸引导、安静准备） |
| 不急于离开 | 灵修结束后提供"留白"时间，而非立即跳转 |

#### Keller/Owen 三层递进框架

```text
        灵修引导的三层递进

        ┌────────────────────────────────────────┐
        │                                        │
        │  1. Mind（理智）                       │
        │     - 理解经文的含义                    │
        │     - AI 提供背景、词汇解释、串珠       │
        │     - 「这节经文说什么？」              │
        │                                        │
        ├────────────────────────────────────────┤
        │                                        │
        │  2. Heart（心灵）                      │
        │     - 感受真理的分量                    │
        │     - AI 引导反思：「这对我意味着什么？」│
        │     - 让真理触动情感                    │
        │                                        │
        ├────────────────────────────────────────┤
        │                                        │
        │  3. Affections（与神相遇）             │
        │     - 从真理进入与神的亲密              │
        │     - AI 退场，引导用户进入安静祷告     │
        │     - 「我现在正在与这位神同在」        │
        │     （AI 边界：这一层 AI 只能引导，    │
        │      真正的相遇是圣灵的工作）           │
        │                                        │
        └────────────────────────────────────────┘
```

#### 场景设计

| 场景 | 设计要点 |
| ---- | -------- |
| 灵修开始 | 3-5秒的"进入"动画/呼吸引导，营造"进入圣所"的仪式感 |
| 灵修过程 | Mind → Heart → Affections 三层递进，逐步引导进入安静 |
| 灵修结束 | 提供"安静留白"选项（1-3分钟），而非立即显示其他内容 |
| AI交互 | 在第三层（Affections）AI主动退场，留出"受控缺口" |

#### 文案示例

| 场景 | 正确设计 |
| ---- | -------- |
| 灵修入口 | "预备进入神的同在" |
| 灵修完成 | "你刚才与神同在了一段时间。" |
| 安静留白提示 | "你可以继续安静片刻。神仍在这里。" |

### 1.4 圣灵边界与 AI 五重边界（Ferguson）

**神学核心**：圣灵是唯一的保惠师（Paraclete）。AI永远只能是辅助，真正的光照、感动、更新是圣灵的工作。

**AI 的独特风险**：AI 与之前所有媒介技术有一个根本区别——**AI 可以模拟位格性互动**。印刷术传递文本，广播传递声音，但它们不会被误认为「一个人」。AI 可以对话、回应情感、给出看似个性化的「牧养」——这意味着它有被用户当作圣灵或牧者替代品的独特风险。微读以五重边界制度化防护（详见 [wd-ministry-and-technology-theology.md](./wd-ministry-and-technology-theology.md) §12）。

#### AI 五重边界

| 边界类型 | 定义 | 实施措施 | 神学来源 |
|---------|-----|---------|---------|
| **权威边界** | AI 永远是"查经与操练的助手"，不是启示来源或牧者替代 | 明确声明局限性；引导用户回到经文本身 | 唯独圣经 |
| **处方边界** | 心理危机、婚姻暴力、邪教/异端、极端神学等高风险主题需"拒答/转介/求助" | 高风险主题检测 + 强制转介 + 可追溯审计 | 爱人如己 |
| **教会边界** | 鼓励用户把关键议题带回真实群体（小组/牧者），让 AI 成为"把人带回教会"的工具 | 关键决策时主动提醒寻求牧者帮助 | Billings D5 |
| **圣灵边界** | AI 只能"指向"，不能"施行"——真正的光照、感动、更新是圣灵的工作 | 在 Mind→Heart→Affections 的第三层，AI 主动退场，引导用户进入与神独处的安静 | **Ferguson** |
| **塑形边界** | 技术从不"中性"——它会放大人心的方向。AI 有将用户塑造成"依赖外部生成、追求快捷结论"的风险 | 定期评估用户是否从"依赖AI"转向"依赖经文与圣灵"；鼓励深度思考而非快速答案 | Smith |

#### 末世论边界（Vos + Ferguson）

AI 运行在"已然-未然"的末世论张力中：

| 维度 | AI 可以做的 | AI 不能做的 | 神学来源 |
|-----|-----------|-----------|---------|
| 真理指引 | 指向基督已经成就的救恩 | 替代圣灵在个人生命中的光照 | Ferguson |
| 身份确认 | 提醒信徒在基督里的身份 | 宣告个人的得救确据 | 唯独圣灵 |
| 苦难中的同在 | 引向哀歌诗篇与基督的受苦 | 解释神在特定苦难中的"目的" | Gaffin 十架神学 |
| 末世问题 | 呈现不同解经立场 | 预测或断言末世事件细节 | 末世论谦卑 |

**核心原则**：AI 是指向基督的工具，不是基督的替代。**圣灵是唯一的 Paraclete（保惠师）**，AI 永远只能是辅助。

#### UI/UX 设计原则

| 原则 | 具体实现 |
| ---- | -------- |
| AI不代言神 | AI永远不说"神对你说..." |
| 引向经文 | AI的回答必须指向经文本身，而非替代经文 |
| 主动退场 | 在关键时刻AI主动"后退"，让用户与神独处 |

#### AI退场时机

| 时机 | 信号 | AI行为 |
| ---- | ---- | ------ |
| 进入默想 | 用户完成Mind/Heart层后 | "接下来是你与神独处的时间。我会安静等待。" |
| 祷告开始 | 用户开始自由祷告 | 停止主动提示，仅保留基础功能 |
| 深度反思 | 用户表达需要安静 | "好的，我会安静陪伴。需要时随时呼唤我。" |
| 情感高峰 | 检测到用户情绪波动大 | 减少文字输出，提供安静空间 |

#### 禁止表达

| 禁止 | 原因 |
| ---- | ---- |
| "神对你说..." | 僭越圣灵的启示工作 |
| "这是神的旨意" | 无权宣告个人的具体旨意 |
| "神要你..." | 将AI意见包装为神的命令 |
| "圣灵感动我告诉你..." | 冒充圣灵的工作 |

### 1.5 Lovelace「首要要素/次要要素」框架

**核心洞见**（Lovelace *Dynamics of Spiritual Life*）：灵命更新需要区分**首要要素**（primary elements）和**次要要素**（secondary elements）。

**首要要素**——福音的核心恩典：
1. **称义确据**：知道自己在基督里被接纳，不再被定罪
2. **成圣动力**：持续被圣灵转化，活出新人的生命
3. **圣灵内住**：神的灵真实住在信徒里面
4. **属灵争战权柄**：在基督里胜过仇敌

**次要要素**——福音的具体实践：
- 读经、祷告、禁食、团契
- 敬拜、服事、见证
- 属灵操练的各种形式

```text
                 「首要要素」与「次要要素」的动力学

            ┌─────────────────────────────────────┐
            │    首要要素（福音的核心恩典）         │
            │                                     │
            │  ┌──────────┐  ┌──────────┐         │
            │  │ 称义确据  │  │ 成圣动力  │         │
            │  └──────────┘  └──────────┘         │
            │  ┌──────────┐  ┌──────────┐         │
            │  │ 圣灵内住  │  │ 争战权柄  │         │
            │  └──────────┘  └──────────┘         │
            └─────────────┬───────────────────────┘
                          │
                    自然流出 ↓
                          │
            ┌─────────────┴───────────────────────┐
            │    次要要素（操练的具体实践）         │
            │                                     │
            │  读经 · 祷告 · 禁食 · 敬拜           │
            │  团契 · 服事 · 见证 · 奉献           │
            └─────────────────────────────────────┘
```

**Lovelace 的核心论点**：**当首要要素健康运作时，次要要素自然流出；当次要要素被强调而首要要素萎缩时，教会陷入律法主义或形式主义。**

#### 产品设计含义

| 反模式（首要要素萎缩） | 正确模式（首要要素优先） |
|---------------------|----------------------|
| 「你需要每天读经」（次要要素先行） | 「你在基督里被接纳——你渴望读经是这身份的自然流露」（首要要素先行） |
| 「坚持打卡就会成长」 | 「回到福音根基，操练自然发生」 |
| 游戏化激励（外在动机） | 身份确认（内在动机流出） |

#### 触发条件

- 任何与用户习惯养成、成就系统相关的功能，首先检查是否先呈现了「首要要素」
- 当用户表现出属灵挣扎时，首先问：「你知道自己在基督里的身份吗？」（首要要素检测）

### 1.6 Lovelace「sarx 强迫性有机体」概念

Lovelace 对肉体（σάρξ, sarx）的分析：**sarx 不仅是个别的犯罪行为，而是一种深嵌于人格结构中的力量系统**——一个「强迫性有机体」。

| 传统理解 | Lovelace 的范式转换 |
|---------|-------------------|
| 罪主要是外在行为问题 | 罪是深嵌人格中的**力量结构** |
| 对策是意志力抵抗和行为规范 | 对抗它需要首要要素的全面运作 |
| 「打地鼠」式的罪行为管理 | 先确认身份，再处理行为 |

#### 产品设计含义

- 任何「打地鼠」式的罪行为管理设计都将失败
- 不是「你需要更努力」，而是「回到福音根基」
- 习惯引擎的目标不是「改正行为」，而是「帮助用户从首要要素出发」

### 1.7 Smith「习惯塑形」框架（新礼仪）

**核心命题**（Smith *You Are What You Love*）：

> 人不是首先思想的存在（homo sapiens），而是首先欲望的存在（homo liturgicus）。你的身份不是由你相信什么决定，而是由你爱什么决定。

| 传统人观 | Smith 的人观 |
|---------|-------------|
| 人是「思想的容器」 | 人是「欲望的动物」 |
| 知识决定行为 | 爱（终极关怀）决定行为 |
| 教育 = 输入信息 | 教育 = 塑造习惯 |
| 改变始于头脑 | 改变始于心——经过身体 |

#### 「新礼仪」（Counter-Liturgy）设计原则

数字产品本身就是「礼仪」——每天打开 app 的仪式在塑造用户成为什么。习惯引擎应该是**基督教的「新礼仪」**——对抗注意力经济的掠夺性礼仪。

| 世俗礼仪（注意力经济） | 习惯引擎的新礼仪 |
|---------------------|----------------|
| 消费主义的「你值得更多」 | 恩典宣告：「你在基督里已经拥有一切」 |
| 即时满足的期待 | 在等待中学习盼望（教会年历节奏） |
| 攀比的等级秩序 | 见证：「神在你生命中的工作」 |
| 原子化的自我 | 指向真实群体：「与基督和彼此联合」 |

**具体产品场景中的新礼仪**（详见 [wd-ministry-and-technology-theology.md](./wd-ministry-and-technology-theology.md) §11）：

| 场景 | 世俗礼仪（注意力经济） | 微读的新礼仪 |
|------|---------------------|-------------|
| 开屏 | 「你落后了 3 天」→ 焦虑 | 「恩典每早晨都是新的」→ 安息 |
| 灵修完成 | 「做得好！+50 积分」→ 成就感依赖 | 「神的话语已临到你」→ 指向恩典 |
| 打卡中断 | 「连续记录已重置」→ 羞耻 | 「欢迎回来。你的身份从未改变」→ 身份确认 |
| AI 交互 | 持续生成更多内容 → 消费惯性 | 第三层主动退场 → 为沉默腾出空间 |

#### 留存与深度的张力

需要诚实承认：微读的产品内部同时存在**留存导向的设计**（打卡、徽章、完成度百分比）和**深度导向的设计**（AI 退场、安息机制、留白）。两者在持续角力。新礼仪不是一个已经实现的成就，而是一个需要在每次设计评审中重新赢得的选择——关键问题始终是：**当留存指标和深度设计冲突时，哪个赢？**

#### 塑形风险评估（每季度检视）

| 评估问题 | 健康信号 | 风险信号（需降低游戏化元素） |
|---------|---------|--------------------------|
| 用户完成灵修后关注什么？ | 经文内容、生命应用 | 积分增加、等级提升 |
| 用户断更后的情绪？ | 恩典叙事，平静回归 | 焦虑、羞耻、强迫补回 |
| 用户讨论的话题？ | 经文洞见、祷告事项 | 比较等级、炫耀徽章 |
| 用户依赖什么？ | 经文与圣灵的光照 | 应用的提醒与激励 |

### 1.8 Whitney「圣经摄取五重方法」

**核心框架**（Whitney *Spiritual Disciplines for the Christian Life*）：

「圣经摄取」不仅是「读」，而是**五种递进方式**——听、读、研、背、默：

```text
                     圣经摄取的五重递进

           听道 ←─→ 读经 ←─→ 研经 ←─→ 背经 ←─→ 默想
             │        │        │        │        │
          被动接收   主动阅读  深入挖掘  内化于心  反复咀嚼
             │        │        │        │        │
           水面      水下      深海      融入     消化
```

**默想是最高层次**——不是额外的操练，而是将其他四种方式的果实**内化**。

> Whitney 引用 Spurgeon：「默想就像『牛反刍』——把吃下去的草再嚼一遍，直到完全消化。」

#### 产品功能与五重方法的对应

| 摄取方式 | 核心要义 | 产品功能对应 | Layer 0 对齐 |
|---------|---------|-------------|-------------|
| **听** | 被动接收——水面 | 语音朗读、有声圣经、讲道 | 神的话语进入 |
| **读** | 主动阅读——水下 | 读经计划、每日灵修 | 与神话语相遇 |
| **研** | 深入挖掘——深海 | Bible Space 研经、注释、串珠 | 更深认识神 |
| **背** | 内化于心——融入 | WeMemo 背诵模块、间隔重复 | 神的话刻在心版 |
| **默** | 反复咀嚼——消化 | 默想引导、Lectio Divina | **练习进入神的同在** |

#### 设计原则

- 避免「信息过载」设计——与其提供更多内容，不如帮助用户**深入咀嚼**
- 默想模块应该是核心——帮助用户从「读过」到「内化」
- 每层都指向 Layer 0：不是「完成任务」，而是「与神相遇」

| **灵魂** | 作为整合中心 | 恢复与神的连接——每日「进入神同在」的操练 |

### 1.10 祷告功能神学设计（Keller + Ferguson）

#### 神学定位

**Keller 祷告双重性质**（*Prayer*）：

> 祷告既是**对话**（conversation），也是**相遇**（encounter）。

| 对话维度 | 相遇维度 |
|---------|---------|
| 与神交谈 | 与神同在 |
| 言语的交换 | 位格的接触 |
| 理性参与 | 情感参与 |
| 可以学习的技巧 | 需要培养的关系 |

两者都是必要的：只有对话而没有相遇，祷告变成任务；只有相遇而没有对话，祷告变成模糊的神秘主义。

| 维度 | 神学内涵 | 产品设计方向 |
|-----|---------|-------------|
| **Telos 维度** | 祷告的终极目的不是「获得回应」，而是**与神相遇** | 引导从「祈求」进入「同在」 |
| **居所维度** | 圣灵住在信徒里面（林前6:19），祷告是回应这一同在 | 提醒「圣灵已在你里面」 |
| **末世维度** | 祷告是「在世俗时间里撕开裂缝，让永恒入侵当下」 | 提前支取将来世代的权能 |
| **职分维度** | 祷告是行使君尊祭司的职分 | 用户是祭司，不是消费者 |

#### Keller/Owen 三阶段祷告框架

```text
        AI 辅助祷告的三层递进

        ┌────────────────────────────────────────┐
        │                                        │
        │  1. Mind（理智）                       │
        │     - 理解经文的含义                    │
        │     - AI 提供祷告框架（ACTS、集祷文）   │
        │     - 「这段经文教导我什么关于神？」    │
        │                                        │
        ├────────────────────────────────────────┤
        │                                        │
        │  2. Heart（心灵）                      │
        │     - 感受真理的分量                    │
        │     - AI 引导：「这对你意味着什么？」   │
        │     - 让真理触动情感                    │
        │                                        │
        ├────────────────────────────────────────┤
        │                                        │
        │  3. Affections（与神相遇）             │
        │     - 从真理进入与神的亲密              │
        │     - AI 退场，留出「受控缺口」         │
        │     - 用户用自己的声音祷告              │
        │     （Ferguson 边界：真正相遇是圣灵工作）│
        │                                        │
        └────────────────────────────────────────┘
```

#### Whitney 圣经祷告法

**核心方法**：以经文为材料祷告——
1. **读一段经文**（通常是诗篇或新约书信）
2. **在每一节停下来**，将经文内容转化为祷告
3. **继续读下一节**，重复这个过程

**产品功能**：「圣经祷告」引导模块——

| 步骤 | AI 角色 | 用户角色 |
|-----|--------|---------|
| 呈现经文 | 朗读/显示一节经文 | 倾听/阅读 |
| 祷告提示 | 提供基于经文的祷告方向 | 用自己的话祷告 |
| 留白 | **静默等待**（受控缺口） | 自由祷告 |
| 下一节 | 继续呈现下一节 | 重复过程 |

#### AI 辅助祷告的神学边界（Ferguson）

**"框架与填充"模式**：AI 的角色是"礼仪助产士"，而非"上帝聊天机器人"。

| 组件 | AI 的角色 | 用户的角色 | 神学边界 |
|-----|---------|-----------|---------|
| 刚性框架 | 提供礼仪结构（如集祷文） | 接受真理锚点 | AI 可以提供框架 |
| 受控缺口 | 创造填充空间，**不替用户祷告** | 自己填补缺口 | AI 不替代用户 |
| 献祭完成 | 引导、确认 | 用自己的声音行使祭司职分 | 用户是祭司 |

**核心边界**（Ferguson）：AI 可以引导，但**真正的相遇是圣灵的工作**。Guided Prayer 中，AI 说话时间 ≤ 用户说话/静默时间。

## 第二部分：指示语气vs命令语气文案规范

### 2.1 核心原则

**指示-命令模式（Indicative-Imperative）**：圣经的结构是先宣告身份（指示），再呼召行动（命令）。产品文案应遵循同样的顺序。

| 语气类型 | 定义 | 比例目标 |
| -------- | ---- | -------- |
| 指示语气 | 宣告用户的身份、恩典、神的工作 | ≥60% |
| 命令语气 | 呼召用户行动、提出要求 | ≤40% |

#### 三个方法论承诺

指示-命令模式背后是微读产品设计的三个方法论承诺（详见 [wd-ministry-and-technology-theology.md](./wd-ministry-and-technology-theology.md) §7）：

1. **身份先于行动。** 产品先宣告用户在基督里的身份，再呼召操练。正确：「神的恩典今天仍为你预备」。错误：「记得每天灵修！」
2. **恩典先于努力。** 先看神做了什么，再问我该回应什么。复活不是证明，而是新创造的开始；联合不是一条教义，而是已经发生的事实。
3. **真实先于舒适。** 信徒同时活在「已然」和「未然」之中。产品设计允许挣扎、允许安息——不是虚假的胜利主义，也不是绝望的失败主义。

### 2.2 文案结构模板

**标准模板**：
```
[指示语气：身份/恩典宣告]
[过渡：因此/所以/基于此]
[命令语气：行动呼召]
```

**示例**：
```
指示：你在基督里是新造的人，旧事已过，都变成新的了。
过渡：因此，
命令：让神的话语继续更新你的心意。

指示：神的恩典每早晨都是新的。
过渡：今天，
命令：你可以在这里与祂相遇。
```

### 2.3 场景文案对照表

| 场景 | 错误（命令先行） | 正确（指示先行） |
| ---- | ---------------- | ---------------- |
| **每日提醒** | "记得每天灵修！" | "神的恩典今天仍为你预备。这里有安静的空间等着你。" |
| **打卡断了** | "你落后了，快赶上！" | "你在基督里的身份不因此改变。让我们重新开始。" |
| **祷告引导** | "你应该为...祷告" | "神已经在听。你可以把...带到他面前。" |
| **读经完成** | "做得好！继续努力！" | "神的话语已经临到你。这是他的礼物。" |
| **新用户引导** | "开始你的属灵成长计划" | "欢迎来到这里。神的话语已为你预备好了。" |
| **连续完成** | "连续30天！太棒了！" | "30天的同行——不是你的坚持，是恩典的托住。" |
| **目标设定** | "设定你的灵修目标" | "神已赐你新生命。让我们一起在祂的话语中成长。" |
| **习惯养成** | "坚持21天养成习惯" | "你已在基督里被更新。每天的读经是活出你所是。" |

### 2.4 禁止句式

| 禁止句式 | 原因 | 替代方案 |
| -------- | ---- | -------- |
| "如果你...神就会..." | 条件式恩典，暗示恩典需要"赚取" | "神已经...因此你可以..." |
| "你需要更努力" | 强化行为主义，忽视身份根基 | "神的力量在你软弱中显为完全" |
| "真正敬虔的人会..." | 设立人为标准，制造比较心理 | "你是神所爱的儿女" |
| "你还不够..." | 否定已然的恩典 | "你在基督里已经完全" |

### 3.4 恩典审计与三层看守机制

九大检验是功能级的设计对齐工具。在其之上，微读通过**三层看守机制**防止使命被商业压力无声侵蚀（详见 [wd-ministry-and-technology-theology.md](./wd-ministry-and-technology-theology.md) §4, §8；[wd-2026-strategy-plan.md](./wd-2026-strategy-plan.md) §1.7, §1.10.3）：

| 机制 | 频率 | 核心功能 |
|-----|------|---------|
| **反 Goodhart 原则** | 持续 | 当量化指标与定性用户反馈/恩典审计矛盾时，后者优先 |
| **恩典审计** | 每季度 | 独立于 OKR Review，回答下方四个核心问题 |
| **属灵成果感知** | 每季度 | 承认「不可测量层」——生命改变和与神相遇无法被数据捕捉；数据活跃但故事贫瘠时视为预警信号 |

#### 恩典审计四个核心问题

每季度独立于 OKR Review 进行：

1. 「我们为了提升 D7 做的事，是帮助用户遇见神还是操控行为？」
2. 「恩典设计是真正的恩典还是包装成恩典的行为激励？」
3. 「安息机制是否真让用户感到被允许休息？」
4. 「行为代理指标还能代表背后的属灵价值吗？」

#### 三层指标模型

| 层级 | 内容 | 使用方式 |
|-----|------|---------|
| **可测量层** | 行为数据（DAU、D7、完成率等） | 驱动日常决策 |
| **可观察层** | NPS / 用户访谈 / 定性反馈 | 每月校验 |
| **不可测量层** | 生命改变 / 与神相遇 | 每季度恩典审计间接感知 |

**当可测量层与不可测量层矛盾时，后者优先。**

## 第四部分：用户旅程与画像

### 4.1 用户属灵旅程六阶段模型

> [!NOTE]
> 此模型是**理解工具**，帮助产品团队理解用户多样性，**不是**判断谁更属灵的等级制度。

```
           属灵旅程六阶段模型

   阶段1        阶段2        阶段3        阶段4
   探索         初信         学习         服事
   Seeking      Beginning    Learning     Serving
      │            │            │            │
      └────────────┴────────────┴────────────┘
                       │
                       ▼
                   「墙」
              (The Wall)
             信仰危机/黑夜
                       │
                       ▼
      ┌────────────────────────────────────┐
      │                                    │
   阶段5                              阶段6
   深化                                联合
   Deepening                          Union
```

**各阶段产品策略**：

| 阶段 | 核心需求 | 产品策略 | 避免 |
|-----|---------|---------|-----|
| **探索期** | 理解福音、解答疑问 | 入门内容、低门槛体验 | 术语堆积、强迫承诺 |
| **初信期** | 基础教义、习惯建立 | 引导式灵修、身份强调 | 复杂操练、过高期待 |
| **学习期** | 系统学习、深入理解 | 深度读经、神学内容 | 仅停留在知识层 |
| **服事期** | 服事神学、避免耗竭 | 服事资源、休息提醒 | 忽视内在生命 |
| **「墙」** | 被听见、十架神学 | 苦难陪伴、指向牧者 | 廉价安慰、过度正面化 |
| **深化/联合** | 默观、深度反思 | 安静资源、导师支持 | 过多干扰 |

### 4.2 神学敏感度四种用户画像

| 画像 | 典型描述 | 核心需求 | 产品策略 | 神学风险 |
|-----|---------|---------|---------|---------|
| **探索者** | 对基督教好奇但尚未信主 | 理解福音、解答疑问 | 入门路径、通俗语言 | 被术语吓走 |
| **成长者** | 已信主1-5年，积极成长 | 系统学习、稳定习惯 | 结构化路径、群体连接 | 律法主义 |
| **深耕者** | 信主多年，稳定属灵生活 | 突破瓶颈、内在更新 | 深度资源、休息提醒 | 属灵疲劳 |
| **挣扎者** | 正经历信仰危机或苦难 | 被听见、不急于解决 | 允许叹息、指向牧者 | 廉价安慰 |

## 第五部分：快速参考卡

### 5.1 设计决策快速检查

**问自己四个问题**：

1. **身份问题**：这个设计是在帮助用户"活出已是"还是"变成什么"？
2. **语气问题**：文案是先宣告身份还是先要求行动？
3. **目的问题**：这个功能指向"与神相遇"还是仅仅"完成任务"？
4. **看守问题**：我们是在帮助用户走向神的同在，还是在无意中成为了一个精致的、被神学语言包装的注意力捕获系统？（详见 [wd-ministry-and-technology-theology.md](./wd-ministry-and-technology-theology.md) §16）

### 5.2 常见反模式速查

| 反模式 | 表现 | 修正方向 |
| ------ | ---- | -------- |
| 律法主义 | 强调必须做到X才能得到Y | 先宣告恩典，再呼召回应 |
| 成功神学 | 只展示成功案例，回避挣扎 | 为失败和挣扎留出空间 |
| 攀比激励 | "看看别人完成了多少" | 改为"见证神在你生命中的工作" |
| 数字崇拜 | 过分强调打卡天数、完成率 | 强调质量和相遇，而非数量 |
| AI替代 | AI试图扮演牧者或圣灵角色 | 明确AI边界，引导回到真实群体 |

### 5.3 恩典导向的 UX 设计原则

| 原则 | 具体做法 | 神学根据 |
|-----|---------|---------|
| **允许失败** | 断了打卡不惩罚，提供重新开始的恩典路径 | 恩典超越律法 |
| **允许瓦器破裂** | 为苦难、哀歌、挣扎提供空间，不强行「正面化」 | Gaffin 十架神学 |
| **尊重节奏** | 不在休息时间推送，允许「安息日模式」 | Kapic 有限性神学 |
| **强调身份** | 每次互动都从「你是谁」开始，而非「你要做什么」 | Gaffin 确定性成圣 |
| **指向群体** | 功能设计引导用户进入小组和教会 | Lovelace 群体人格 |

**失败场景的恩典设计**：

| 场景 | 传统设计（律法倾向） | 恩典设计 |
|-----|-------------------|---------|
| 连续打卡中断 | 「你的连续记录已重置」 | 「欢迎回来。恩典每天都是新的」 |
| 长时间未使用 | 「你已经X天没有灵修了」 | 「我们为你保留了这个安静的空间」 |
| 读经计划未完成 | 「你落后了X天」 | 「不论你在哪里，都可以从这里开始」 |
| 祷告时间不足 | 「今日祷告时间：2分钟（低于平均）」 | 「每一分钟的安静都是宝贵的」 |

### 5.4 紧急情况处理

**检测到用户处于危机状态时**：

1. **立即降低AI活跃度**
2. **推送牧者/心理咨询热线联系方式**
3. **使用哀歌模式响应**
4. **记录用于后续跟进**（需用户同意）

## 附录B：技术在救赎历史中的位置

微读的设计决策需要放在一个更大的背景中理解：技术在救赎历史中不是被动存在的——它被圣灵主动征用来服务圣道的传播。每一代的技术都执行了同样的四重功能：**创造空间、保存传承、延伸触达、使人参与**。从比撒列建造会幕→文士传承圣道→保罗书信→印刷术→微读，共同模式是：**技术被圣灵征用来创造神与人相遇的新空间。** 微读站在这条线上——不是开创一种新的关系，而是在数字时代延续一个古老的模式。

完整论述详见 [wd-ministry-and-technology-theology.md](./wd-ministry-and-technology-theology.md) §10。
