## 第一部分：战略背景与OKR原则

### 1.1 2026战略主题

> **2026主题：内容为王，归属为锚**
>
> - **内容为王**：在LLM/Agent时代，高质量内容是最难复制的护城河——通用AI可以生成任何界面，但无法凭空创造独家版权内容
> - **归属为锚**：复杂流程、精细体验、群体归属感构成的产品不易被通用Agent替代——用户留下不是因为功能，而是因为在这里有关系、有节奏、有身份

### 1.2 北极星指标：WESU

> **唯一北极星：WESU（Weekly Engaged Scripture Users）— 每周至少完成 1 次高质量圣经参与会话的独立用户数（跨全部微读产品去重）。**

「高质量圣经参与会话」= A + B（两类各至少满足一项）：
- **A（圣道接触）**：阅读 ≥1 章经文 / 聆听 ≥3 分钟 / 背诵练习
- **B（停留回应）**：停留 ≥2 分钟 / 笔记 / 高亮 / AI 提问 / 祷告

> WESU 的完整定义、跨产品适用性和反 Goodhart 保护，详见 [wd-2026-strategy-plan.md](./wd-2026-strategy-plan.md) §1.6。

**健康指标与可持续性护栏**（原四维北极星降格）：

| 定位 | 指标 | 2026目标 | 与 WESU 的因果关系 |
| :--- | :--- | -------: | :------- |
| **增长健康指标** | 全球（排除中国大陆）周活增长率 | +100% | WESU 的漏斗顶部输入 |
| **留存健康指标** | 新用户第7日留存率 | ≥40% | 确保新用户有机会成为 WESU |
| **可持续性护栏** | 年度自营收入 | $830,000 | 使命延续的财务保障 |
| **组织能力指标** | AI驱动人均产出效率 | +100% | 基石层，支撑团队交付 |

> **收入口径说明**：本文档自营收入为 PRS+Ilumine $490K + 书城 $150K + 奉献 $180K + SaaS $10K = $830K，不含 Major Donors。含 Major Donors 的组织总收入 $3.33M 见战略规划 §1.6.4 收入结构与责任归属。

> **因果优先级**：**WESU > 留存 > 增长 > 效率 > 收入**。WESU 是瓦器的核心功能指标；留存是增长的根基；增长是自营收入的前提；效率是杠杆。

**WESU 深度队列**（原行为北极星）：
> WESU 中每周 ≥2 次高质量会话的用户占比（目标 ≥50%），作为 WESU 的质量维度追踪。

### 1.4 OKR设计原则

1. **使命驱动**：每个 Objective 直接连接"与神同住"的终极目的
2. **激励优先**：表述回答"为什么"而非仅"做什么"
3. **结果导向**：KR以用户行为/业务结果衡量，而非功能上线
4. **神学正直**：AI相关KR包含信任与准确性约束
5. **挑战但可达**：70%完成率为成功标准（Google OKR哲学）
6. **先行-滞后分层**：每个O区分"结果KR"（滞后指标）和"先行指标"（监控看板），避免KR膨胀
7. **解决方案→结果**：每个KR从"要做什么事"出发，回答"如果做成了，会有什么可衡量的结果"

### O1: 筑起 AI 时代最深的内容护城河

**战略对应**：优先战略1 — 内容护城河
**WESU 贡献**：全部（内容是三重飞轮的起点，为 WESU 提供圣道接触的素材基础）

> **使命陈述**：通用AI可以在几秒内生成任何界面，但无法凭空创造独家的、经过神学审校的属灵内容。内容是微读在LLM时代最不可替代的资产——丰富授权版权库、建立自有版权库、构建PGC内容生态，让这个护城河深到任何竞争者都无法跨越。内容护城河不仅是版权库的广度，更是属灵操练资源的深度——祷告指南、读经计划、查经材料、灵修引导是习惯引擎（O2）和 WESU 的素材命脉。公版权经典中蕴藏着丰富的查经与灵修指导资源，应有意识地纳入翻译管线。

- **慕道友/非信徒**（5-10%）：护教入门/信仰探索/见证故事 → BS探索模式的内容来源

### O2: 让灵修成为呼吸，让归属不可替代

**战略对应**：优先战略2 — 习惯与群体
**WESU 贡献**：WESU 的核心驱动力（习惯引擎直接提升 WESU + 留存确保用户有机会成为 WESU）

### O3: 打造全球最可信赖的 AI Native 属灵助手

**战略对应**：优先战略3 — AI原生产品
**WESU 贡献**：为更多用户打开深度圣经参与的新入口（AI 引导查经直接贡献 WESU）

> **使命陈述**：Bible Space 的差异化不是"更好的LLM"，而是通用AI做不到的三件事：独家内容**拿不到**、真实关系**给不了**、全流程体验**拼不起来**。以"可信赖"为根基，让AI成为指向经文的谦卑助手——圣灵是唯一的保惠师，AI只能引导，真正的相遇是圣灵的工作。

> **说明**：Bible Space等AI类产品2026年聚焦产品验证，**暂不设置收入目标**，收入贡献纳入2027规划。

## 第四部分：OKR管理机制

### 4.1 评分标准

采用Google OKR评分体系：

| 分数 | 含义 | 说明 |
| :--- | :--- | :--- |
| 0.0 - 0.3 | 未达成 | 需要复盘根因，调整策略 |
| 0.4 - 0.6 | 部分达成 | 合理进展，继续努力 |
| 0.7 - 1.0 | 达成/超额达成 | 目标设定可能过于保守 |

### 4.2 复盘节奏

| 频率 | 格式 | 参与者 | 内容 | 时长 |
| :--- | :--- | :----- | :--- | :--- |
| **周五演示** | Live Demo + 讨论 | 全体 | 各小队演示 1 个可工作成果或洞察；AI 工具最佳实践分享；阻塞即时暴露 | 60min |
| **月度对齐** | 结构化 Review | 领导层+小队负责人 | KR 进度评分（0-1 scale）；先行指标审查；风险/阻塞升级；资源再分配决策 | 90min |
| **关键事项即时拉齐** | Ad-hoc | 相关方 | P0 问题、跨小队阻塞、资源冲突——不等月度会议。≤24h 响应目标 | 按需 |
| 季度 OKR 复盘 | 半日工作坊 | 全体 | OKR 评分；假设验证；飞轮健康检查；防御健康度；恩典审计；下季度调整 | 半日 |
| 年度 OKR 总结 | 全日工作坊 | 全体 | 年度评分；2027 规划输入 | 全日 |

> 核心变化：周会→**周五演示**（展示可工作成果，非幻灯片汇报）；新增**即时拉齐**机制（关键事项不等节奏会议）。

**飞轮健康检查（季度 OKR 复盘项）**：

### 4.3 神学校准机制

每季度进行"神学对齐评估"——检视各项指标是否真正服务于"帮助信徒活出与复活基督联合的生命"。

#### 4.3.1 代理指标偏离检测

| 代理指标 | 可能的偏离 | 神学校准问题 |
| :------- | :--------- | :----------- |
| 周活增长 +100% | 追求数字而非生命 | 这些用户是否更像基督？ |
| D7 留存 ≥40% | 优化行为而非身份 | 留存是因为习惯还是因为遇见神？ |
| 收入 $830K | 商业化侵蚀使命 | 收入是否服务于更多人的属灵益处？ |
| 效率 +100% | 用 AI 替代真实关系 | 效率是否释放更多时间用于关系建造？ |

#### 4.3.2 九大神学检验问题

每个产品功能上线前，需通过以下综合检验：

| # | 检验问题 | 神学来源 | 通过标准 |
| :- | :------- | :------- | :------- |
| 1 | **Union Test** | Gaffin 确定性成圣 | 是否帮助用户"活出已发生的联合"而非"制造联合"？ |
| 2 | **Spirit Test** | Ferguson 圣灵论 | 是否把圣灵的角色交还给圣灵，而非AI？ |
| 3 | **Embodiment Test** | Kapic 有限性神学 | 是否尊重身体、节奏、休息与有限性？ |
| 4 | **Church Test** | Lovelace 万人祭司 | 是否加强教会与群体？圣礼界线是否清晰？ |
| 5 | **Telos Test** | Boersma 荣福直观 | 是否指向"与神同住/见神"的终极目的？ |
| 6 | **Cross Test** | Gaffin 十架神学 | 是否允许"瓦器破裂"？是否在苦难中彰显复活？ |
| 7 | **Liturgy Test** | Smith 习惯塑形 | 这个功能在塑造什么欲望？会不会成为新偶像？ |
| 8 | **创造测试** | Moo 创造神学 | 是否有意识地包含创造/管家维度？ |
| 9 | **同住测试** | Morales 门廊礼仪 | 是否帮助用户体验"进入神的同在"？ |

### 4.4 恩典审计与属灵成果感知

#### 4.4.1 恩典审计（每季度，独立于OKR Review）

每季度进行独立的恩典审计，回答五个核心问题：

1. "我们为了提升 WESU 做的事，是帮用户遇见神还是操控行为？"
2. "恩典设计是真恩典还是包装的行为激励？"
3. "安息机制是否真让用户感到被允许休息？"
4. "行为代理指标还能代表属灵价值吗？"
5. "WESU 是否仍在测量我们认为它在测量的东西——用户与圣道的深度相遇？"

#### 4.4.2 恩典健康度监控指标（独立看板，不进入OKR KR）

| 指标 | 健康区间 | 监控频率 |
| :--- | :------- | :------- |
| 安息选择率 | 10-25% | 每月 |
| 恩典叙事效果（断更回归后积极反馈率） | ≥60% | 每月 |
| 灵修后安静时长 | 2-5分钟 | 每月 |
| AI退场率（Mind→Heart→Affections第三层） | ≥50% | 每月 |
| 用户自述属灵收获（有实质描述比例） | ≥40% | 每季度 |

#### 4.4.3 三层指标架构

- **可测量层**（行为数据）→ 日常决策。**北极星 WESU** 是可测量层的主指标
- **可观察层**（NPS/CSAT/访谈）→ 每月校验：WESU 增长是否伴随用户属灵深度的正面反馈？
- **不可测量层**（生命改变）→ 每季度恩典审计：WESU 代理的行为是否仍然指向真实的属灵成长？

> **当可测量与可观察/不可测量矛盾时，后者优先。**

#### 4.4.4 属灵成果感知机制（每季度，恩典审计的一部分）

代理指标无法测量"用户是否更深认识基督、是否在操练中遇见神、生命是否改变"。以下机制用于间接感知这些不可测量的成果：

1. **用户故事收集**：每季度通过App内问卷邀请活跃用户分享"这个季度微读对你属灵生命的影响"（开放式文本，非打分）。目标：每季度≥50份有效回复。
2. **生命见证精选**：从收集的故事中精选5-10个代表性见证，在团队季度Review中共读——让团队直接听到用户声音，而非仅看数据。
3. **指标校准检验**：将见证内容与同期量化指标对比——如果数据显示"WESU上升、灵修频次增加"但用户故事中缺乏属灵深度描述，视为预警信号，触发恩典审计深度讨论。
4. **反向案例关注**：特别关注"数据上不活跃但描述了深度属灵成长"的用户——他们可能提示我们的活跃定义过窄。

### 4.5 假设验证检查点

战略规划基于一系列隐含假设，需在季度Review中系统性验证。

**检查点一：Q1 Review**
- 假设验证：留存是首要杠杆、AI效率翻倍可行性、B2B $490K Pipeline覆盖、巴西Q1就绪
- 决策输出：Q1资源分配调整，由领导层在Q1 Review中确认

**检查点二：Q2 Review**
- 假设验证：BS神学差异化验证、小组对留存的因果贡献（原KR2.7降级至此——验证标准：小组用户月留存差≥20pp，断更回归率差≥10pp）
- 决策输出：H2计划调整，由领导层在Q2 Review中确认

### 4.8 品位把关原则：Coach 优先而非 Gatekeeper

> 产品体验与方向需要统一品位标准，但不应形成「等人拍板」的依赖模式。

| 阶段 | 旧模式（Gatekeeper） | 新模式（Coach） |
| :--- | :--- | :--- |
| 需求定义 | 小队定义→等审批 | 前期方法辅导（设计方法论培训）；小队独立应用 |
| 设计评审 | 单人逐一审核每个设计 | 小队内同行评审 + 异步设计批评频道；Coach 仅在被请求时或 P0 变更时介入 |
| 质量验收 | 上线前单点签字 | 小队持有发布权，依据质量检查清单（源自神学设计指南 + UX 标准）自主决策 |
| 升级 | 一切经过一人 | 仅在涉及 AI 神学输出、祷告内容、牧养引导等神学敏感功能时，走神学治理委员会审查 |

**具体执行**：
1. **前期方法论**：月度设计方法论分享会，覆盖神学设计指南的应用、品质标准
2. **异步评审**：建立设计批评频道，鼓励但不强制跨小队异步反馈
3. **发布权下放**：小队负责人持有发布权限，通过质量检查清单自主决策
4. **神学例外**：涉及 AI 神学输出、祷告内容、牧养引导的功能仍需神学治理委员会审查
5. **事后学习**：月度对齐会中加入「设计回顾」环节，复盘已发布功能，提取模式

### 4.9 OKR 覆盖度规则

> **OKR ≠ 全部工作的唯一框架**。OKR 覆盖各小队 70-80% 的主力精力，保留 20-30% 的必要弹性空间。

| 弹性空间类型 | 示例 | 治理方式 |
| :--- | :--- | :--- |
| 突发机会 | 意外出现的合作伙伴、紧急用户请求 | 小队负责人酌情决策，周五演示中汇报 |
| 技术债务 | 重构、基础设施维护、测试覆盖 | 独立追踪（技术债指数），不占 OKR 指标 |
| 学习成长 | AI 技能发展、神学研读、行业会议 | 归入人才发展（KR5.3）框架 |
| 跨小队支援 | 帮助其他小队解决阻塞 | 通过跨小队阻塞解决 SLA 追踪 |
| 创新实验 | 当前 OKR 范围外的小规模实验 | 必须限时（≤2周），在周五演示分享发现 |

**规则**：
1. 小队负责人确保 70-80% 对齐比率
2. 非 OKR 工作超过 30% → 在月度对齐中升级讨论再平衡
3. 非 OKR 工作**可见但不隐形**——在周五演示和月度 Review 中汇报
4. 若非 OKR 活动证明有价值，可在季度 Review 中正式纳入为新 KR
